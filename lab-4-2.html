Lab 4.2 — Symmetric Encryption with OpenSSL and AES

Network Security hands-on lab based on the Lab 1.1 virtual environment. Students use OpenSSL to encrypt and decrypt files with AES, verify confidentiality through ciphertext inspection, compare encrypted and plaintext files, and document safe key/password handling practices.

1. Lab Purpose

This lab introduces symmetric encryption using OpenSSL and AES. Students will create a plaintext file, encrypt it with AES-256-CBC using a passphrase-based key derivation option, inspect the ciphertext, decrypt it, and verify that the decrypted file matches the original.

Kali-Lab ───── Host-only / Private Network ───── Ubuntu-Target
Main goal: Students should understand that symmetric encryption protects confidentiality, while hashing verifies integrity. These two functions are related but not the same.

2. Learning Objectives

  1. Explain the purpose of symmetric encryption.
  2. Explain the difference between encryption and hashing.
  3. Create a sample plaintext file for encryption testing.
  4. Encrypt a file using OpenSSL and AES-256-CBC.
  5. Use salt and PBKDF2 options during passphrase-based encryption.
  6. Inspect ciphertext and explain why it should not be readable.
  7. Decrypt a file using the correct passphrase.
  8. Demonstrate failed decryption with an incorrect passphrase.
  9. Verify that the decrypted file matches the original file using hashes and diff.
  10. Document OS-specific differences for Windows, Intel Mac, Apple Silicon Mac, Kali, and Ubuntu.

3. Lab Scenario

A security analyst needs to protect a small configuration file before transferring it across a lab network. The analyst encrypts the file on Ubuntu-Target, transfers the encrypted file to Kali-Lab, decrypts it with the correct passphrase, verifies the result, and documents what happens when an incorrect passphrase is used.

6. Safety Rules

  1. Use only files created for this lab.
  2. Do not encrypt, copy, or submit real passwords, private keys, student records, personal data, or sensitive documents.
  3. Do not use the lab passphrase for any real account.
  4. Do not submit the actual passphrase in screenshots or reports.
  5. Do not use bridged networking.
  6. Keep both VMs on the isolated host-only/private network.
  7. Do not claim encryption provides integrity by itself; verify integrity separately.
Passphrase rule: For this lab, use a temporary class-only passphrase. Do not reuse a personal password.

7. Key Concepts

Symmetric Encryption

Symmetric encryption uses the same secret key, or a key derived from the same passphrase, to encrypt and decrypt data. Anyone with the correct key can decrypt the file.

Encryption vs. Hashing

FunctionPurposeReversible?Example Command
EncryptionProtect confidentialityYes, with the correct key/passphraseopenssl enc -aes-256-cbc
HashingVerify integrityNosha256sum file

AES

Advanced Encryption Standard, or AES, is a widely used symmetric encryption algorithm. This lab uses AES-256-CBC for a simple file-encryption exercise through OpenSSL.

Salt and PBKDF2

A salt helps make repeated encryption outputs different, and PBKDF2 strengthens passphrase-based key derivation by applying repeated computation.

Part 4 — Create a Plaintext File on Ubuntu

cd ~/Lab4-2-Encryption/plaintext
cat > confidential_config.txt <<'EOF'
System Name: Ubuntu-Target
Lab: 4.2 Symmetric Encryption
Classification: Lab Training Data Only
Database Host: db.internal.lab
Admin Contact: student@example.local
Security Note: This is not real sensitive information.
EOF
cat confidential_config.txt

Part 5 — Generate a Baseline Hash of the Plaintext

cd ~/Lab4-2-Encryption/plaintext
sha256sum confidential_config.txt
sha256sum confidential_config.txt > ~/Lab4-2-Encryption/hashes/confidential_config_plaintext.sha256

Part 6 — Inspect the Plaintext File

file ~/Lab4-2-Encryption/plaintext/confidential_config.txt
xxd ~/Lab4-2-Encryption/plaintext/confidential_config.txt | head
cat ~/Lab4-2-Encryption/plaintext/confidential_config.txt

Observation: The plaintext file is readable with cat, and readable text appears in the hex output.

Part 7 — Encrypt the File with AES-256-CBC

On Ubuntu-Target, run the command below. When prompted, enter a temporary lab passphrase. Do not use a real password.

openssl enc -aes-256-cbc -salt -pbkdf2 -iter 100000 \
-in ~/Lab4-2-Encryption/plaintext/confidential_config.txt \
-out ~/Lab4-2-Encryption/encrypted/confidential_config.txt.enc

Verify that the encrypted file exists:

ls -lh ~/Lab4-2-Encryption/encrypted/
file ~/Lab4-2-Encryption/encrypted/confidential_config.txt.enc

Part 8 — Inspect the Ciphertext

cat ~/Lab4-2-Encryption/encrypted/confidential_config.txt.enc
xxd ~/Lab4-2-Encryption/encrypted/confidential_config.txt.enc | head
sha256sum ~/Lab4-2-Encryption/encrypted/confidential_config.txt.enc \
> ~/Lab4-2-Encryption/hashes/confidential_config_ciphertext.sha256

Observation: The ciphertext should not be readable as normal text. The hex output should look different from the plaintext file.

Part 9 — Transfer the Encrypted File and Baseline Hash to Kali

# On Kali-Lab
scp YOUR-UBUNTU-USERNAME@UBUNTU-IP-ADDRESS:/home/YOUR-UBUNTU-USERNAME/Lab4-2-Encryption/encrypted/confidential_config.txt.enc ~/Lab4-2-Encryption/received/
scp YOUR-UBUNTU-USERNAME@UBUNTU-IP-ADDRESS:/home/YOUR-UBUNTU-USERNAME/Lab4-2-Encryption/hashes/confidential_config_plaintext.sha256 ~/Lab4-2-Encryption/hashes/
ls -lh ~/Lab4-2-Encryption/received/
ls -lh ~/Lab4-2-Encryption/hashes/

Part 10 — Decrypt the File on Kali

On Kali-Lab, use the same passphrase used during encryption:

openssl enc -d -aes-256-cbc -pbkdf2 -iter 100000 \
-in ~/Lab4-2-Encryption/received/confidential_config.txt.enc \
-out ~/Lab4-2-Encryption/decrypted/confidential_config_decrypted.txt

Verify the decrypted file:

cat ~/Lab4-2-Encryption/decrypted/confidential_config_decrypted.txt
file ~/Lab4-2-Encryption/decrypted/confidential_config_decrypted.txt

Part 11 — Verify the Decrypted File Against the Original Hash

The original hash file refers to confidential_config.txt. Create a hash-check file that points to the decrypted file:

ORIGINAL_HASH=$(cut -d ' ' -f1 ~/Lab4-2-Encryption/hashes/confidential_config_plaintext.sha256)
echo "$ORIGINAL_HASH  ~/Lab4-2-Encryption/decrypted/confidential_config_decrypted.txt" > ~/Lab4-2-Encryption/hashes/check_decrypted.sha256
sha256sum -c ~/Lab4-2-Encryption/hashes/check_decrypted.sha256
sha256sum -c ~/Lab4-2-Encryption/hashes/check_decrypted.sha256 \
> ~/Lab4-2-Encryption/comparisons/decrypted-integrity-check.txt

Expected result: confidential_config_decrypted.txt: OK.

Part 12 — Demonstrate Failed Decryption with an Incorrect Passphrase

On Kali-Lab, intentionally use the wrong passphrase when prompted:

openssl enc -d -aes-256-cbc -pbkdf2 -iter 100000 \
-in ~/Lab4-2-Encryption/received/confidential_config.txt.enc \
-out ~/Lab4-2-Encryption/decrypted/wrong_passphrase_output.txt

Save a note describing the failure:

cat > ~/Lab4-2-Encryption/comparisons/wrong-passphrase-result.txt <<'EOF'
A wrong passphrase was tested.
Expected result: decryption failed or produced unusable output.
The real passphrase was not recorded in this evidence file.
EOF
Do not submit screenshots showing the passphrase.

Part 13 — Compare Plaintext, Ciphertext, and Decrypted Files

On Kali-Lab, compare visible file types and bytes:

file ~/Lab4-2-Encryption/received/confidential_config.txt.enc
file ~/Lab4-2-Encryption/decrypted/confidential_config_decrypted.txt
xxd ~/Lab4-2-Encryption/received/confidential_config.txt.enc | head
xxd ~/Lab4-2-Encryption/decrypted/confidential_config_decrypted.txt | head

Save the comparison:

{
  echo "Encrypted file type:"; file ~/Lab4-2-Encryption/received/confidential_config.txt.enc
  echo; echo "Decrypted file type:"; file ~/Lab4-2-Encryption/decrypted/confidential_config_decrypted.txt

  echo; echo "Decrypted hex sample:"; xxd ~/Lab4-2-Encryption/decrypted/confidential_config_decrypted.txt | head
} > ~/Lab4-2-Encryption/comparisons/plaintext-ciphertext-decrypted-comparison.txt

Part 14 — Re-Encrypt the Same Plaintext and Compare Ciphertexts

Because a salt is used, encrypting the same plaintext twice should produce different ciphertext files.

# On Ubuntu-Target
openssl enc -aes-256-cbc -salt -pbkdf2 -iter 100000 \
-in ~/Lab4-2-Encryption/plaintext/confidential_config.txt \
-out ~/Lab4-2-Encryption/encrypted/confidential_config_second.enc

sha256sum ~/Lab4-2-Encryption/encrypted/confidential_config.txt.enc ~/Lab4-2-Encryption/encrypted/confidential_config_second.enc \
> ~/Lab4-2-Encryption/hashes/two-ciphertexts-comparison.sha256

Observation: The two ciphertext hashes should usually be different because encryption used a random salt.

15. Troubleshooting Guide

ProblemLikely CauseFix
Kali cannot ping UbuntuWrong IP or VM network mismatchVerify ip -brief addr and host-only/private network mode.
openssl command not foundOpenSSL package missingInstall OpenSSL if allowed, or use a VM image that includes it.
Decryption fails with correct passphraseTypo, wrong file, or options do not match encryption commandUse the same algorithm, PBKDF2, and iteration count used during encryption.
Decrypted hash check failsWrong output file, wrong baseline hash, or failed decryptionRe-run decryption and recreate the hash-check file carefully.
Ciphertext appears strange in terminalEncrypted file is binary-like dataThis is expected. Use xxd or file for inspection.
scp failsWrong username, IP, path, or SSH issueVerify SSH login from Kali to Ubuntu first.
Two encrypted files are differentSalt creates different encryption outputThis is expected when using -salt.

16. Instructor Notes

Core lesson: Encryption protects confidentiality, but it does not replace integrity verification. Students should pair encryption evidence with hash verification to prove the decrypted file matches the original.

Recommended Teaching Sequence

  1. Review encryption vs. hashing.
  2. Create a plaintext file on Ubuntu.
  3. Generate the SHA-256 baseline hash.
  4. Encrypt the file with OpenSSL AES-256-CBC using salt and PBKDF2.
  5. Inspect ciphertext and compare it to plaintext.
  6. Transfer encrypted file to Kali.
  7. Decrypt using the correct passphrase.
  8. Verify decrypted file with SHA-256.
  9. Test a wrong passphrase and discuss the result.
  10. Complete the final report.
