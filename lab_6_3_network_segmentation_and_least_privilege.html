1. Lab Purpose

The purpose of this lab is to teach students how to enforce network segmentation and least privilege using pfSense firewall rules. Students will define which systems should communicate, which ports should be allowed, which traffic should be blocked, and how to verify that the final rule set matches the intended security policy.

Main goal: Students should understand that least privilege in networking means allowing only required traffic between zones, not allowing broad access simply because hosts are in the same lab.

2. Learning Objectives

  1. Explain network segmentation and least privilege in practical firewall terms.
  2. Identify network zones, assets, trust levels, and required traffic flows.
  3. Create pfSense aliases for readable and maintainable firewall rules.
  4. Replace broad inter-zone access with specific allow and block rules.
  5. Use rule order to enforce specific permissions before broader deny rules.
  6. Allow LAN access to a specific DMZ web service while blocking other LAN-to-DMZ traffic.
  7. Block DMZ-initiated access to the LAN.
  8. Optionally allow limited DMZ outbound access for DNS, HTTP, and HTTPS only.
  9. Verify firewall policy with ping, curl, nmap, state tables, and logs.
  10. Document segmentation evidence, least-privilege rule decisions, exceptions, and remaining risk.

3. Lab Scenario

A small organization has created a DMZ for a web server, but early firewall rules are too broad. The security team wants to enforce least privilege: the internal LAN should reach only the DMZ web service, the DMZ should not initiate connections into the LAN, and outbound access from the DMZ should be limited. Your task is to design, apply, test, and document a least-privilege segmentation policy.

6. Safety Rules

  1. Use only the local virtual lab environment.
  2. Do not create firewall rules that expose pfSense WebGUI to the WAN or DMZ.
  3. Do not create broad WAN-to-LAN rules.
  4. Do not scan public or non-lab systems.
  5. Take a pfSense snapshot before changing rules.
  6. Do not delete default rules unless instructed; disable or move rules only with instructor approval.
  7. Use logging on lab block rules so rule behavior can be verified.
  8. Document every rule added, moved, disabled, or removed.
Important: Least privilege can break connectivity if rules are too restrictive or ordered incorrectly. Make one change at a time and test after each change.

7. Key Concepts

Network Segmentation

Network segmentation separates systems into zones, such as LAN, DMZ, and WAN. Each zone has a different trust level and requires different firewall rules.

Least Privilege

Least privilege means users, systems, and networks receive only the access needed to perform their function. In firewall design, this means specific source, destination, protocol, and port rules instead of broad allow-any rules.

Default Deny

A default deny approach blocks traffic unless an explicit rule allows it. pfSense has an implicit deny at the end of each interface rule set, but broad allow rules can override the security intent if placed above more specific blocks.

Rule Order

pfSense evaluates rules from top to bottom on the interface where traffic enters. The first matching rule wins. Rule order is critical for least privilege.

Aliases

Aliases are named objects for hosts, networks, or ports. They make firewall rules easier to understand and maintain.

Part 7 — Create LAN-to-DMZ Non-Web Block Rule

On Firewall > Rules > LAN, add this block rule below the web allow rule but above any broad LAN allow rule:

FieldRecommended Value
ActionBlock or Reject
InterfaceLAN
ProtocolAny
SourceLAN_NET
DestinationDMZ_NET
DescriptionBlock LAN to DMZ except explicit allowed services above
LoggingEnable
Why this works: The web allow rule must be above this block rule. The block rule then prevents other LAN-to-DMZ traffic from falling through to a broad LAN allow rule below.

16. Instructor Notes

Core lesson: This lab turns the DMZ topology into a least-privilege policy exercise. Students should prove both positive access and negative access: what is allowed and what is blocked.
  1. Confirm students have working Lab 6.2 topology before starting.
  2. Consider providing a starting pfSense configuration snapshot.
  3. Require rule-order screenshots because rule order is the most common error.
  4. Tell students whether ICMP should be allowed or blocked between LAN and DMZ.
  5. Tell students whether DMZ outbound internet access is allowed.
  6. Use the risk and exception register to discuss why some broad rules remain in real environments.
