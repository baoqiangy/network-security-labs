1. Lab Purpose

The purpose of this lab is to help students practice alert triage. Students will examine alerts from Wazuh, Suricata, Linux logs, or instructor-provided alert cards. They will determine whether each alert is expected, benign, false positive, suspicious, or needs escalation. The lab emphasizes evidence-based decision making instead of reacting to alert titles alone.

Main goal: Students should learn that triage is the process of asking: What happened? Who or what was involved? How severe is it? Is it expected? What evidence supports the decision? What should happen next?

2. Learning Objectives

  1. Explain the purpose of alert triage in security operations.
  2. Identify key alert fields such as timestamp, source, destination, agent, rule ID, signature, severity, and event message.
  3. Use local logs, Wazuh dashboard events, and Suricata alerts as supporting evidence.
  4. Classify alerts as benign expected activity, false positive, suspicious activity, or escalation candidate.
  5. Assign severity based on event context, affected asset, user activity, and potential impact.
  6. Correlate network IDS evidence with endpoint logs where available.
  7. Create a timeline for related events.
  8. Document triage decisions in an analyst worksheet.
  9. Recommend next steps such as monitor, tune rule, investigate, contain, or escalate.
  10. Explain OS-specific differences in alert sources and log evidence.

3. Lab Scenario

You are working as a junior security analyst. During your shift, several alerts appear in the monitoring environment. Some alerts are expected lab activity, some are harmless but noisy, and some require closer investigation. Your job is to review the evidence, assign a triage decision, document your reasoning, and prepare a concise analyst handoff note.

6. Safety Rules

  1. Use only the local lab environment or instructor-provided alert data.
  2. Do not generate new attack traffic unless specifically assigned and approved.
  3. Do not run brute-force attacks, exploit tools, malware, or denial-of-service tests.
  4. Do not scan public IP addresses, school networks, classmates' systems, or production systems.
  5. Do not submit passwords, tokens, private keys, or dashboard credentials.
  6. Sanitize screenshots that show sensitive usernames, internal secrets, or tokens.
  7. Document uncertainty honestly. Do not overstate a finding as confirmed compromise without evidence.
  8. Focus on triage, evidence, and reasoning.
Important: Alert triage is not the same as proving compromise. A triage decision should clearly distinguish confirmed facts, assumptions, and recommended next steps.

7. Key Concepts

Alert

An alert is a notification generated by a monitoring system when an event matches a detection rule, signature, threshold, or suspicious pattern.

Triage

Triage is the first-stage review of an alert to determine priority, likely meaning, evidence quality, and next action.

Severity

Severity describes potential impact and urgency. Severity should be based on context, not only the alert title.

False Positive

A false positive is an alert that appears suspicious but does not represent harmful or policy-violating activity after review.

Benign True Positive

A benign true positive is an alert that correctly identifies an event, but the event was expected or authorized, such as a lab test.

Escalation

Escalation means passing an alert to a higher-level analyst, incident response team, or instructor because the event may require further investigation or action.

Triage CategoryMeaningExample
Benign expected activityAlert is accurate but expectedStudent-generated SSH login during lab
False positiveAlert does not represent the suspected riskIDS rule matches harmless test string
SuspiciousAlert has concerning indicators but needs more evidenceUnexpected failed login from unknown source
EscalateAlert may indicate real compromise or high riskMultiple unauthorized sudo attempts plus suspicious file changes

11. OS-Specific Differences

Ubuntu-Agent / Ubuntu-Target

Ubuntu provides local endpoint evidence through authentication logs, journald, auditd, Wazuh agent logs, and application logs.

sudo tail -n 100 /var/log/auth.log
sudo journalctl --since "60 minutes ago" --no-pager
sudo ausearch -k KEY -i
sudo tail -n 100 /var/ossec/logs/ossec.log

Kali-Lab

Kali is the controlled event source and evidence collection workstation. It can generate safe SSH, curl, and selected-port test activity for local lab triage.

ssh USER@UBUNTU-IP "hostname && whoami"
curl -I http://UBUNTU-IP
nmap -sT -Pn -p 22,80,443 UBUNTU-IP
scp -r USER@UBUNTU-IP:/home/USER/Lab7-4-AlertTriage ./ubuntu-evidence
