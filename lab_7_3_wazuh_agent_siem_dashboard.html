1. Lab Purpose

The purpose of this lab is to introduce endpoint monitoring with the Wazuh agent and alert review with the Wazuh SIEM dashboard. Students will verify a Wazuh manager/dashboard, install or enroll a Linux agent, generate controlled endpoint events, and use the dashboard to confirm that the endpoint is visible and producing security data.

Main goal: Students should understand the difference between host logging, agent-based telemetry, and SIEM dashboard analysis.

2. Learning Objectives

  1. Explain the role of Wazuh as an endpoint monitoring and SIEM/XDR platform.
  2. Identify the major Wazuh components: manager/server, indexer, dashboard, and agent.
  3. Access the Wazuh dashboard and locate agent deployment instructions.
  4. Install and enroll a Wazuh agent on Ubuntu-Target.
  5. Verify agent connectivity from the endpoint, Wazuh server, and dashboard.
  6. Generate safe security events such as SSH login, sudo use, file changes, and optional auditd events.
  7. Review agent status, events, alerts, inventory, and security configuration assessment results.
  8. Use dashboard filters to find events from a specific agent.
  9. Document alert evidence, rule IDs, severity levels, timestamps, and event meaning.
  10. Explain OS-specific differences for Linux, Windows, macOS, Kali, Ubuntu, and Apple Silicon environments.

3. Lab Scenario

A small organization has deployed a Wazuh SIEM server and wants to onboard Linux endpoints for centralized security monitoring. You are assigned to enroll Ubuntu-Target as a monitored endpoint, generate controlled events, confirm that the Wazuh dashboard receives data, and write a short analyst report explaining what the dashboard shows.

6. Safety Rules

  1. Use only the assigned Wazuh server and local lab endpoints.
  2. Do not submit dashboard passwords, API keys, private keys, or reusable credentials.
  3. Do not scan or monitor systems outside the lab.
  4. Generate only safe endpoint events such as one SSH login, one sudo command, and lab file changes.
  5. Do not perform brute-force testing. One optional failed login is enough if the instructor requests it.
  6. Do not install the Wazuh manager on a production system.
  7. Do not expose the Wazuh dashboard to public networks.
  8. Sanitize screenshots if they show passwords, tokens, or sensitive personal data.
Important: SIEM dashboards can contain sensitive hostnames, usernames, IP addresses, alerts, and log details. Treat dashboard evidence as security-sensitive.

7. Key Concepts

SIEM

A Security Information and Event Management system centralizes security data, correlates events, and provides dashboards for monitoring, investigation, and reporting.

Wazuh Agent

The Wazuh agent runs on a monitored endpoint. It collects logs, file integrity events, inventory data, configuration assessment results, and other telemetry and sends them to the Wazuh server.

Wazuh Manager / Server

The Wazuh manager receives agent data, applies decoders and rules, generates alerts, and coordinates endpoint monitoring functions.

Wazuh Indexer

The indexer stores and indexes security data so it can be searched and visualized.

Wazuh Dashboard

The dashboard is the web interface where analysts view agents, alerts, inventory, compliance, vulnerabilities, configuration assessment results, and other security data.

Agent Enrollment

Agent enrollment connects an endpoint to the Wazuh manager and creates the trust relationship needed for the agent to send data.

Use the dashboard-generated command first: Wazuh package URLs and version numbers change. The dashboard usually provides the best command for the server version in use.

Example placeholder pattern only:

sudo systemctl enable wazuh-agent
sudo systemctl start wazuh-agent

11. OS-Specific Differences

Ubuntu-Agent

Ubuntu is the monitored Linux endpoint in this lab. It uses the Wazuh Linux agent and local logs such as /var/log/auth.log, journalctl, and optional auditd logs.

sudo systemctl status wazuh-agent --no-pager
sudo tail -n 100 /var/ossec/logs/ossec.log
sudo grep -i sudo /var/log/auth.log | tail
sudo journalctl --since "30 minutes ago" --no-pager

Kali-Lab

Kali is mainly used to generate safe remote events and collect evidence. Kali can also be enrolled as a Wazuh agent if the instructor wants to monitor both systems.

ssh USER@UBUNTU-AGENT-IP "hostname && whoami"
curl -k -I https://WAZUH-DASHBOARD-IP
scp -r USER@UBUNTU-AGENT-IP:/home/USER/Lab7-3-Wazuh ./ubuntu-evidence

Wazuh-Server

The Wazuh server may be a dedicated VM, official virtual appliance, instructor-managed server, or all-in-one installation.

sudo systemctl status wazuh-manager --no-pager
sudo systemctl status wazuh-indexer --no-pager
sudo systemctl status wazuh-dashboard --no-pager
sudo /var/ossec/bin/agent_control -l

15. Troubleshooting Guide

ProblemLikely CauseFix
Dashboard does not loadWrong IP, service down, certificate warning, firewall issue, or network mismatchPing Wazuh server, test curl -k -I, check dashboard service, and verify network.
Agent installation command failsWrong package type, no internet, version mismatch, or repository issueUse dashboard-generated command for the correct OS/package and ask instructor if offline install is needed.
Agent service not runningInstallation incomplete or configuration errorCheck systemctl status wazuh-agent and /var/ossec/logs/ossec.log.
Agent does not appear in dashboardEnrollment failed, wrong manager IP, blocked ports, or dashboard time delayCheck manager address, agent logs, Wazuh manager status, and firewall rules.
Agent appears disconnectedAgent stopped, network failure, or key/enrollment issueRestart agent and check ossec.log for connection errors.
No SSH/sudo event appearsTime filter wrong, event not collected, or processing delayExpand dashboard time range, filter by agent name, and compare local logs.
FIM event does not appearDirectory not monitored or scan interval delayConfirm syscheck configuration and wait for scheduled scan or restart agent if instructed.
