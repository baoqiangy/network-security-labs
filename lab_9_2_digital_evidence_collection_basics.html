1. Lab Purpose

The purpose of this lab is to introduce students to basic digital evidence collection practices in a safe classroom environment. Students will collect evidence from a simulated incident using only dummy files and lab logs, calculate hashes, document metadata, preserve copies, complete a chain-of-custody record, and package evidence for review.

Main goal: Students should understand that evidence collection is not simply copying files. It requires scope, documentation, integrity verification, careful handling, and clear reporting.

2. Learning Objectives

  1. Explain the purpose of digital evidence collection during incident response.
  2. Distinguish original evidence, working copies, screenshots, logs, and notes.
  3. Create an evidence collection scope and evidence numbering system.
  4. Collect dummy incident files without modifying the original more than necessary.
  5. Record file metadata such as path, size, owner, permissions, and timestamps.
  6. Generate SHA-256 hashes before and after evidence copying.
  7. Collect relevant Linux logs using safe commands.
  8. Capture screenshots as supporting evidence.
  9. Complete an evidence inventory and chain-of-custody form.
  10. Package evidence and verify the final evidence archive hash.
  11. Explain evidence collection differences across Linux, Windows, macOS, cloud, and network devices.

3. Lab Scenario

A user reports suspicious activity on an internal Linux endpoint. A few unusual files have appeared in a shared directory, and authentication logs show recent access activity. You are assigned to collect basic digital evidence for the incident response team. Because this is a classroom lab, all evidence is simulated and stored in a designated lab directory. Your job is to preserve the evidence, document what you collected, verify integrity, and prepare a short evidence collection report.

6. Safety Rules

  1. Use only simulated evidence created for this lab.
  2. Do not collect real personal files, real credentials, private keys, tokens, or sensitive documents.
  3. Do not access systems, accounts, or files without permission.
  4. Do not perform real forensic imaging of disks unless the instructor explicitly assigns a separate controlled exercise.
  5. Do not run malware, exploit tools, or destructive scripts.
  6. Do not alter original evidence after collection begins except where the lab explicitly creates simulated files.
  7. Do not delete evidence files during the lab.
  8. Clearly label all evidence as simulated classroom evidence.
  9. Redact screenshots if they accidentally show sensitive information.
  10. Document any mistake honestly in the evidence handling notes.
Important: This lab teaches defensive evidence handling using dummy data. It does not authorize collection from real systems or people.

7. Key Concepts

Digital Evidence

Digital evidence is information stored or transmitted in digital form that may help explain what happened during an incident. Examples include files, logs, metadata, screenshots, alerts, and configuration records.

Evidence Integrity

Evidence integrity means the evidence has not been altered unexpectedly. Cryptographic hashes such as SHA-256 help show whether a file changed.

Original Evidence vs. Working Copy

Original evidence should be preserved as much as possible. Analysts should work from copies when feasible so they do not accidentally alter the original.

Metadata

Metadata is data about data. For files, metadata includes file size, owner, permissions, path, and timestamps.

Chain of Custody

Chain of custody documents who collected, handled, transferred, stored, or accessed evidence. It supports trust and accountability.

Scope

Scope defines what evidence will be collected and what will not be collected. Clear scope reduces privacy risk and prevents unnecessary collection.

Part 8 — Hash Copied Evidence and Compare

Generate hashes for copied evidence and compare them with the original hashes.

cd ~/Lab9-2-DigitalEvidence/evidence-copies

find . -type f ! -name "evidence-copy-file-list.txt" -print0 | sort -z | xargs -0 sha256sum \
  > ~/Lab9-2-DigitalEvidence/copy-hashes/copied-evidence-sha256.txt

cd ~/Lab9-2-DigitalEvidence

diff -u original-hashes/original-evidence-sha256.txt copy-hashes/copied-evidence-sha256.txt \
  > copy-hashes/original-vs-copy-hash-comparison.txt || true

if [ ! -s copy-hashes/original-vs-copy-hash-comparison.txt ]; then
  echo "PASS: Copied evidence hashes match original evidence hashes." | tee copy-hashes/hash-comparison-result.txt
else
  echo "REVIEW NEEDED: Copied evidence hashes differ from original evidence hashes." | tee copy-hashes/hash-comparison-result.txt
fi

cat copy-hashes/hash-comparison-result.txt
cat copy-hashes/original-vs-copy-hash-comparison.txt

Part 10 — Collect Linux Logs for the Lab Time Range

Collect only relevant logs. Use a short time range and avoid collecting unnecessary personal information.

cd ~/Lab9-2-DigitalEvidence

journalctl --since "2 hours ago" --no-pager > logs/journal-last-2-hours.txt 2>&1
journalctl --since "2 hours ago" --no-pager | grep -Ei "ssh|sudo|session|login|authentication" \
  > logs/journal-auth-related-last-2-hours.txt 2>&1 || true

  sudo tail -n 200 /var/log/auth.log > logs/auth-log-tail-200.txt
else
  echo "/var/log/auth.log not present on this system." > logs/auth-log-not-present-note.txt
fi

sha256sum logs/*.txt > logs/log-evidence-sha256.txt 2>/dev/null || true
ls -lh logs | tee logs/log-directory-listing.txt

11. OS-Specific Differences

Ubuntu / Linux

Linux evidence collection commonly uses file metadata commands, hashes, journald logs, authentication logs, service state, and network state.

sha256sum file.txt
stat file.txt
journalctl --since "2 hours ago" --no-pager
sudo tail -n 200 /var/log/auth.log
ss -tulpen
ps aux

Kali Linux

Kali can act as an evidence review and transfer workstation. It should only collect evidence from systems where collection is authorized.

scp USER@HOST:/path/to/evidence.tar.gz .
sha256sum -c evidence.tar.gz.sha256
tar -tzf evidence.tar.gz | head

15. Troubleshooting Guide

ProblemLikely CauseFix
Hash comparison failsFile changed, extra file included, or paths differCheck source/copy file lists and ensure the same files are hashed.
auth.log does not existDistribution logs authentication elsewhereUse journalctl and create an auth-log-not-present note.
Permission denied reading logsLogs require elevated privilegesUse sudo only if permitted; otherwise document limitation.
Evidence archive is too largeCollected too much dataReduce scope to required simulated evidence and relevant logs only.
Screenshots show sensitive informationTerminal or logs included real dataRetake or redact screenshots before submission.
Package hash verification fails after transferTransfer incomplete or wrong hash fileRe-copy archive and hash file, then verify again.
Chain-of-custody form is incompleteTransfers or handling steps were not recordedAdd collection, storage, transfer, and submission entries.
Students modify original evidence after hashingWorking from original instead of copyDocument the mistake and repeat collection if possible using new simulated evidence.
