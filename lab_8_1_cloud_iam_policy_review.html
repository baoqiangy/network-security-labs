Network Security hands-on lab based on a safe cloud security review workflow. Students examine cloud identities, roles, groups, service accounts, access keys, role assignments, and policy documents to identify excessive privileges, wildcard permissions, stale accounts, missing MFA, risky trust relationships, and least-privilege improvement opportunities.

1. Lab Purpose

The purpose of this lab is to teach students how to review cloud IAM policies from a security analyst perspective. Students will identify risky permissions, evaluate whether access follows least privilege, distinguish user identities from service identities, review trust and assignment relationships, and recommend safer IAM configurations without making destructive changes.

Main goal: Students should learn that cloud security depends heavily on identity. In cloud environments, excessive IAM permissions can be as dangerous as exposed network services.

2. Learning Objectives

  1. Explain the purpose of cloud IAM in protecting cloud resources.
  2. Identify users, groups, roles, service accounts, access keys, and policy assignments.
  3. Review cloud IAM policies for least-privilege alignment.
  4. Identify risky patterns such as administrative access, wildcard actions, wildcard resources, and broad role assignments.
  5. Review service accounts and machine identities for overprivileged access.
  6. Evaluate MFA status, stale credentials, unused accounts, and access key age where available.
  7. Distinguish identity-based policies, resource-based policies, role assignments, and trust policies.
  8. Use a risk rating method to prioritize IAM findings.
  9. Recommend remediation steps without making unauthorized changes.
  10. Compare IAM concepts across AWS, Microsoft Azure, Google Cloud, Linux CLI environments, Windows PowerShell, and browser-based cloud consoles.

3. Lab Scenario

A small organization has recently moved part of its infrastructure to the cloud. The organization created several cloud users, groups, roles, and service accounts quickly during testing. Management is concerned that some accounts may have excessive permissions. You have been asked to perform a read-only IAM policy review and prepare an analyst report that identifies risks and recommends least-privilege improvements.

6. Safety Rules

  1. Use only the cloud account, subscription, project, or policy files assigned by the instructor.
  2. Do not make permission changes unless the instructor explicitly tells you to do so.
  3. Do not create public access to storage, databases, virtual machines, or management interfaces.
  4. Do not create or download long-term access keys unless assigned.
  5. Do not submit secret access keys, private keys, tokens, passwords, or session credentials.
  6. Do not run commands that create billable resources unless instructed.
  7. Do not test access against systems or cloud tenants you do not own or have permission to review.
  8. Use read-only screenshots and redact account IDs, usernames, and sensitive identifiers if required.
Important: IAM review can expose sensitive account structure and privilege relationships. Treat all cloud identity information as confidential lab evidence.

7. Key Concepts

Identity and Access Management

Cloud IAM controls who or what can access cloud resources and what actions they can perform. IAM includes human users, groups, roles, service accounts, managed identities, policies, and trust relationships.

Least Privilege

Least privilege means identities should receive only the permissions required for their assigned function, for only the resources they need, and only under appropriate conditions.

Policy

A policy is a rule document or assignment that grants or denies actions. Policies commonly include principals, actions, resources, effects, and conditions.

Principal

A principal is the user, service account, role, application, or external identity that receives permission or is allowed to assume a role.

Wildcard Permission

A wildcard such as * can mean all actions, all resources, or a broad set of permissions. Wildcards may be appropriate in limited cases, but they require careful justification.

Trust Policy

A trust policy defines who can assume or use a role. Risky trust relationships can allow unintended identities to gain access.

Stale Identity

A stale identity is a user, role, key, or service account that is no longer used but still has permissions. Stale identities increase attack surface.

Part 8 — Review a Policy Document for Risky Patterns

Create a policy review worksheet for one policy. Use this sample risk pattern checklist:

  • Does the policy allow all actions, such as *?
  • Does the policy allow all resources, such as *?
  • Does it grant administrative actions?
  • Does it allow identity management, such as creating users, keys, roles, or policy attachments?
  • Does it allow data exfiltration actions, such as reading all storage objects or secrets?
  • Does it lack conditions such as MFA, source IP, resource tags, or time restrictions?
  • Is the policy attached to a human user, service account, group, or role?
  • Is the scope broader than needed?

Part 17 — Create a Sample Before/After Policy Improvement

cat > ~/Lab8-1-CloudIAM/analysis/sample-before-after-policy-review.txt <<'EOF'
Lab 8.1 Sample Before/After Policy Improvement

Before — Broad Policy Example:
{
  "Effect": "Allow",
  "Action": "*",
  "Resource": "*"
}

Risk:
- Allows all actions on all resources.
- Difficult to justify for routine user or service account access.
- Could allow identity changes, data access, resource deletion, and security control changes.

After — Narrower Example:
{
  "Effect": "Allow",
  "Action": [
    "storage:GetObject",
    "storage:ListBucket"
  ],
  "Resource": [
    "specific-bucket",
    "specific-bucket/*"
  ],
  "Condition": {
    "RequireApprovedContext": "true"
  }
}

Improvement:
- Limits action type.
- Limits resource scope.
- Adds contextual control.
- Supports least privilege.
EOF

Windows PowerShell

Windows PowerShell uses different file paths and archive/hash commands, but can run cloud CLIs if installed.

New-Item -ItemType Directory -Force -Path "$HOME\Lab8-1-CloudIAM"
Get-Content .\policy.json
Select-String -Path .\policy.json -Pattern '"\*"'
Compress-Archive -Path "$HOME\Lab8-1-CloudIAM" -DestinationPath "$HOME\Lab8-1-CloudIAM-Submission.zip" -Force

macOS

macOS Terminal works similarly to Linux for many commands. Some package installation steps differ, and Apple Silicon users should ensure CLI packages support ARM64.

mkdir -p ~/Lab8-1-CloudIAM
cat policy.json
grep -R '"*"' .
shasum -a 256 Lab8-1-CloudIAM-Submission.tar.gz

Browser-Based Cloud Console

The browser console is the safest option when students do not need CLI exports. It is also helpful for screenshots and visual review, but students should avoid capturing secrets or full account identifiers if redaction is required.
