1. Lab Purpose

The purpose of this capstone lab is to bring together the major technical and management skills developed throughout the course. Students will build or document a small defensive network, verify its configuration, collect evidence, evaluate risks, and produce a final professional network defense report.

Main goal: Students should demonstrate that they can move from individual technical tasks to an integrated security defense process: identify assets, reduce exposure, harden systems, monitor activity, validate recovery, plan response, collect evidence, and communicate risk.

2. Learning Objectives

  1. Design or document a small defensive lab network using clear network diagrams and asset inventory.
  2. Verify host, network, firewall, service, and access-control baselines.
  3. Use safe network discovery to identify exposed services in the lab environment.
  4. Apply or document endpoint hardening controls such as updates, users, sudo, SSH, firewall rules, services, permissions, and logging.
  5. Review network segmentation and least-privilege access between zones or hosts.
  6. Collect monitoring evidence from Linux logs, Suricata, Wazuh, firewall logs, or sample alerts.
  7. Perform a safe incident response tabletop using capstone evidence.
  8. Validate backup and restore readiness using dummy files or previous Lab 8.4 evidence.
  9. Create a digital evidence package with hashes and chain-of-custody documentation.
  10. Build a risk register using findings from the capstone environment.
  11. Produce a final network defense audit report with prioritized recommendations.
  12. Present the capstone findings in a concise analyst-style briefing or recorded demonstration.

3. Capstone Scenario

A small organization has asked you to perform a final review of its internal network defense posture. The environment includes at least one Linux endpoint/server, a security workstation, optional pfSense firewall segmentation, optional IDS/SIEM monitoring, and a backup process. Your task is to document the environment, validate key defenses, identify risks, and recommend practical improvements.

Instructor option: The capstone can be performed in the full pfSense/DMZ/SIEM topology, or in a simplified topology with Kali-Lab and Ubuntu-Target only. Students should clearly document which option they used.

6. Safety Rules

  1. Use only the assigned lab environment.
  2. Do not scan public IP addresses, school production networks, classmates’ systems, or systems without permission.
  3. Do not perform exploit testing, brute-force attacks, malware execution, denial-of-service testing, or ransomware testing.
  4. Use safe service discovery only against assigned lab IP addresses.
  5. Use dummy data for backup, restore, evidence, and incident response simulations.
  6. Do not include passwords, private keys, tokens, API keys, or sensitive personal data in submissions.
  7. Do not intentionally weaken systems to create findings unless the instructor explicitly provides a controlled scenario.
  8. Document limitations honestly instead of inventing evidence.
  9. Clearly separate confirmed findings, assumptions, and recommendations.
  10. Keep your risk register and evidence package protected because they describe weaknesses.
Important: This is a defensive capstone. The expected work is validation, documentation, hardening, monitoring, recovery testing, risk analysis, and reporting—not offensive exploitation.

7. Key Concepts

Defense in Depth

Defense in depth uses multiple layers of controls so that a single failure does not expose the entire environment. Layers may include segmentation, firewall rules, endpoint hardening, authentication, monitoring, backups, and incident response.

Security Baseline

A security baseline is a documented starting point for system and network configuration. It helps determine whether a system is configured securely and consistently.

Least Privilege

Least privilege limits users, services, and network flows to only what is required. It applies to accounts, SSH, firewall rules, cloud IAM, containers, and administrative roles.

Monitoring and Detection

Monitoring helps identify suspicious activity using logs, IDS alerts, SIEM dashboards, firewall events, and endpoint activity.

Recovery Readiness

Recovery readiness means important data and systems can be restored in an acceptable time frame and validated after restoration.

Risk-Based Reporting

Risk-based reporting translates technical findings into business priorities by evaluating likelihood, impact, existing controls, recommended actions, residual risk, ownership, and due dates.

sha256sum backups/capstone-backup.tar.gz > checksums/capstone-backup-archive.sha256

tar -xzf backups/capstone-backup.tar.gz -C restore
cd restore/live-data
sha256sum capstone-file.txt > ../../checksums/restored-capstone-file.sha256

cd ~/Lab9-4-NetworkDefenseCapstone/backup-test

15. Troubleshooting Guide

ProblemLikely CauseFix
Too many tools unavailableSimplified environment or time limitUse simplified topology and document limitations clearly.
Nmap scan shows no open portsWrong IP, firewall blocking, or service not runningVerify IP address, check ss -tulpen, and scan only assigned ports.
Cannot access Wazuh or SuricataOptional tools not installed or not runningUse Linux logs or instructor sample alerts and document limitation.
Firewall changes block SSHIncorrect UFW/pfSense ruleUse console access to correct rule; document issue and recovery.
Backup validation failsWrong file restored or checksum path mismatchRecreate dummy backup/restore test and compare correct files.
Risk register entries are vagueFindings lack evidenceTie each risk to evidence from scans, logs, hardening, backup, or tabletop results.
Final report is too technicalNo executive summary or prioritizationAdd business impact, risk level, and practical recommendations.
Evidence archive is missing filesFiles saved on different VM or directoryCopy Ubuntu evidence to Kali and create a final file list before archiving.
Screenshots show secretsTerminal or dashboard displayed sensitive dataRetake or redact screenshots before submission.
