1. Lab Purpose

The purpose of this lab is to introduce Linux user and group access control. Students will create user accounts, assign group membership, configure file ownership and permissions, test access decisions, and document how Linux enforces least privilege.

Main goal: Students should understand that access control is not only about creating users. It requires carefully combining identity, group membership, ownership, permissions, and privilege boundaries.

2. Learning Objectives

  1. Explain Linux user, group, and permission concepts.
  2. Create local user accounts for different roles.
  3. Create groups and assign users to groups.
  4. Use chown and chmod to control file access.
  5. Test file access as different users.
  6. Apply least privilege to shared project folders.
  7. Review sudo group membership and administrative privileges.
  8. Create a safe sudo restriction example using a drop-in sudoers file.
  9. Document a permission matrix and access-control test results.
  10. Explain OS-specific differences across Windows, Intel Mac, Apple Silicon Mac, Kali, and Ubuntu environments.

3. Lab Scenario

A small organization has created a Linux server for three groups: administrators, analysts, and guests. Analysts need access to project files, guests should have very limited access, and only administrators should perform privileged actions. You will configure Linux users and groups to enforce this access model and then test whether it works.

6. Safety Rules

  1. Perform account and permission changes only on the assigned Ubuntu-Target VM.
  2. Do not modify real user accounts on host systems.
  3. Do not delete system accounts or remove your own sudo access.
  4. Do not edit /etc/sudoers directly with a normal text editor; use visudo or a safe drop-in file.
  5. Do not submit passwords or password hashes.
  6. Use only temporary lab users and folders created in this lab.
  7. Keep the lab network host-only/private.
Important: Incorrect account or sudo configuration can lock you out or break administrative access. Follow the commands carefully and keep console access to Ubuntu-Target available.

Part 13 — Create a Safe Limited Sudo Example

Use a limited test command only. This step demonstrates sudo restriction using a safe command. Do not give broad administrator access to lab users.

On Ubuntu-Target, allow admin1 to run only /usr/bin/id with sudo without giving full administrative privileges:

echo "admin1 ALL=(root) NOPASSWD: /usr/bin/id" | sudo tee /etc/sudoers.d/lab31-admin1-id
sudo chmod 440 /etc/sudoers.d/lab31-admin1-id
sudo visudo -cf /etc/sudoers.d/lab31-admin1-id

Test the limited sudo command:

sudo -u admin1 sudo /usr/bin/id

Test a command that should not be allowed:

sudo -u admin1 sudo /usr/bin/apt update

Expected result: admin1 should be allowed to run id but not broad administrative commands such as apt update.

Save evidence:

sudo -u admin1 sudo /usr/bin/id > ~/Lab3-1-AccessControl/sudo-tests/admin1-limited-sudo-id-allowed.txt 2>&1 || true

sudo -u admin1 sudo /usr/bin/apt update > ~/Lab3-1-AccessControl/sudo-tests/admin1-apt-denied.txt 2>&1 || true

sudo cat /etc/sudoers.d/lab31-admin1-id > ~/Lab3-1-AccessControl/sudo-tests/limited-sudo-rule.txt
