1. Lab Purpose

The purpose of this lab is to introduce container security fundamentals using Docker in a local, controlled environment. Students will compare how containers run by default with safer settings such as non-root users, read-only filesystems, dropped capabilities, limited ports, resource controls, and careful secrets handling.

Main goal: Students should understand that containers are not automatically secure. Container security requires secure images, least-privilege runtime settings, careful network exposure, and monitoring.

2. Learning Objectives

  1. Explain the difference between containers, images, Dockerfiles, registries, and runtimes.
  2. Install or verify Docker in a local lab environment.
  3. Run a basic container and inspect its image, process, ports, mounts, and logs.
  4. Identify risky container settings such as root user, privileged mode, exposed ports, bind mounts, and embedded secrets.
  5. Build a simple container image using a Dockerfile.
  6. Modify a Dockerfile to run as a non-root user.
  7. Apply runtime protections such as read-only filesystem, dropped capabilities, no-new-privileges, and resource limits.
  8. Review image metadata and container configuration using docker inspect.
  9. Document container security findings and least-privilege recommendations.
  10. Explain Docker and container differences across Ubuntu, Kali, Windows, macOS, and Apple Silicon systems.

3. Lab Scenario

A development team has started using containers for small web applications. The team can run containers, but they have not reviewed security settings. You are asked to perform a basic container security review, identify risky defaults, create a safer container configuration, and document recommendations for future container deployments.

6. Safety Rules

  1. Use only local lab containers and instructor-approved images.
  2. Do not run unknown images from untrusted registries.
  3. Do not use --privileged except for discussion or instructor-provided screenshots; do not run privileged containers in this lab.
  4. Do not mount sensitive host directories such as /, /etc, /home, /var/run/docker.sock, or SSH key folders.
  5. Do not put real passwords, tokens, API keys, or private keys into Dockerfiles or images.
  6. Do not expose container ports to public networks.
  7. Do not attempt container escape techniques or exploit testing.
  8. Remove lab containers when finished unless the instructor asks you to keep them for grading.
Important: The Docker daemon is powerful. A user who can control Docker often has high control over the host. Treat Docker access as privileged access.

7. Key Concepts

Container

A container is a running instance of an image. It packages an application with its dependencies and runs as an isolated process on the host.

Image

An image is a layered template used to create containers. Images may include application code, libraries, configuration, and metadata.

Dockerfile

A Dockerfile is a text file that defines how an image is built. It includes instructions such as FROM, COPY, RUN, USER, EXPOSE, and CMD.

Root in a Container

A process running as root inside a container may still create risk, especially if the container has broad capabilities, dangerous mounts, or weak runtime isolation.

Capabilities

Linux capabilities divide some root privileges into smaller units. Dropping unnecessary capabilities reduces what a container can do.

Read-Only Filesystem

A read-only container filesystem limits write access inside the container. Temporary writable locations can be added only when needed.

Secrets Handling

Secrets should not be hard-coded in Dockerfiles, image layers, environment files committed to source control, or screenshots. Use approved secret management methods.

Part 9 — Build a Non-Root Image

Create a hardened Dockerfile that runs as a non-root user. This example uses a Python web server so the non-root configuration is easy to see.

cd ~/Lab8-2-ContainerSecurity/build/lab82-web

cat > Dockerfile.nonroot <<'EOF'
FROM python:3.12-alpine

RUN addgroup -S appgroup && adduser -S appuser -G appgroup
WORKDIR /app
COPY index.html /app/index.html
RUN chown -R appuser:appgroup /app

USER appuser
EXPOSE 8000
CMD ["python", "-m", "http.server", "8000"]
EOF

docker build -f Dockerfile.nonroot -t lab82-web-nonroot:1.0 .
docker image ls lab82-web-nonroot:1.0 | tee ~/Lab8-2-ContainerSecurity/hardened/lab82-web-nonroot-image.txt

Part 14 — Review Secrets Handling Mistakes

Create a sample bad Dockerfile snippet without real secrets. Do not use real credentials.

cat > ~/Lab8-2-ContainerSecurity/secrets-review/bad-secrets-example.txt <<'EOF'
Bad Example — Do Not Use Real Secrets

FROM alpine
ENV API_KEY="fake-demo-key-do-not-use"
COPY private-key.pem /app/private-key.pem
CMD ["sh"]

Why this is bad:
- Secrets can remain in image layers.
- Secrets may appear in image history.
- Secrets may be exposed to anyone with image access.
- Secrets may be captured in logs or screenshots.

Safer ideas:
- Use a secret manager.
- Inject secrets at runtime through approved secure methods.
- Avoid committing secrets to source control.
- Rotate credentials if a secret is accidentally placed in an image.
EOF

11. OS-Specific Differences

Ubuntu

Ubuntu is often the easiest Linux environment for Docker Engine labs. Students may use sudo docker unless the docker group is configured.

sudo apt update
sudo apt install -y docker.io
sudo systemctl enable --now docker
sudo docker version

Kali Linux

Kali can run Docker, but it is usually better as a security testing VM. For this lab, Kali may be used if Docker is installed or as a client to test container ports on another VM.

sudo systemctl status docker --no-pager
docker version
curl -I http://CONTAINER-HOST-IP:8080

15. Troubleshooting Guide

ProblemLikely CauseFix
docker command not foundDocker is not installed or not in PATHInstall Docker or use instructor-provided container VM.
Permission denied running DockerUser lacks Docker permissionsUse sudo docker on Linux or follow instructor permission setup.
Docker daemon not runningService stopped or Docker Desktop closedStart Docker service or open Docker Desktop.
Port already in useAnother service/container uses the host portUse a different host port such as 8081, 8082, or 8084.
Image pull failsNo internet or registry blockedUse instructor-provided image tar file or preloaded image.
Container exits immediatelyCommand failed or application errorCheck docker logs and docker ps -a.
Hardened container fails with read-only filesystemApplication needs writable pathAdd a limited --tmpfs mount for required temporary writes.
jq not foundJSON tool not installedSkip jq extraction or install jq if instructor permits.
Apple Silicon image errorImage architecture mismatchUse a multi-architecture image or instructor-provided ARM64-compatible image.
Cleanup failsContainer name does not exist or is already removedDocument result and continue.
