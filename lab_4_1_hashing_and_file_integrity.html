1. Lab Purpose

The purpose of this lab is to teach students how hashing supports file integrity verification. Students will create known-good files, calculate hash values, build a baseline manifest, modify files, detect unauthorized changes, and explain how hashing supports integrity in cybersecurity operations.

Main goal: Students should understand that a hash does not prevent a file from changing, but it provides strong evidence that a file has changed.

2. Learning Objectives

  1. Define hashing and explain its role in integrity verification.
  2. Generate MD5, SHA-256, and SHA-512 hashes for lab files.
  3. Explain why SHA-256 and SHA-512 are preferred over MD5 for integrity evidence.
  4. Create a file integrity baseline using sha256sum.
  5. Verify a file against a saved checksum.
  6. Detect modified, added, and deleted files using a hash manifest.
  7. Compare hashes before and after file transfer between Ubuntu-Target and Kali-Lab.
  8. Create an integrity incident note based on hash mismatch evidence.
  9. Document hash evidence in a professional report.
  10. Explain OS-specific differences for Windows, macOS Intel, macOS Apple Silicon, Kali, and Ubuntu.

3. Lab Scenario

You are a junior security analyst responsible for monitoring the integrity of important configuration and evidence files on a Linux server. Your supervisor asks you to create a known-good hash baseline, verify the files later, and document any changes. You must also prove that a file transferred from Ubuntu-Target to Kali-Lab did not change during transfer.

6. Safety Rules

  1. Use only files and folders created for this lab.
  2. Do not hash, copy, modify, or submit real sensitive files.
  3. Do not modify operating system files or production configuration files.
  4. Do not submit private keys, passwords, password hashes, or personal files.
  5. Do not treat MD5 as strong security protection; it is included only for comparison and legacy awareness.
  6. Keep the lab network host-only/private.
  7. Document all changes made during the lab.
Important: This lab demonstrates file integrity safely. It does not involve malware, cracking, or modifying system-critical files.

7. Key Concepts

Hashing

A hash function takes input data and produces a fixed-length output called a digest or hash value. A small change in the input should produce a very different hash value.

File Integrity

File integrity means a file remains complete, accurate, and unchanged except by authorized actions. Hashing helps verify whether a file is still the same as a known-good version.

Checksum Manifest

A checksum manifest stores hash values and file names. It can later be used to verify whether files still match their original values.

15. Troubleshooting Guide

ProblemLikely CauseFix
sha256sum -c says file not foundCommand is being run from the wrong directoryRun it from the directory where the files are located or use correct relative paths.
Verification fails unexpectedlyFile content changed, line endings changed, or wrong baseline usedCompare file content and confirm you are using the correct baseline manifest.
Diff shows many changesManifest order changedUse find ... | sort before hashing directory files.
md5sum not foundMinimal system image missing toolUse SHA-256 and document that MD5 comparison was unavailable.
scp failsWrong username, IP, path, or SSH issueVerify SSH access and the exact evidence file path.
Transferred archive verification failsHash file contains a path that does not match local filenameEdit the hash file carefully or run sha256sum lab4-1-files-evidence.tar.gz and compare manually.
