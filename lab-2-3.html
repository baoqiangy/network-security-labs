1. Lab Purpose

This lab introduces HTTP service testing in a safe virtual environment. Students will start a temporary web service, test it from another VM, inspect HTTP response headers and status codes, observe the difference between local and remote access, and review the security implications of plaintext HTTP traffic.

Kali-Lab ───── Host-only / Private Network ───── Ubuntu-Target
Main goal: Students should understand how a simple HTTP service is exposed, how to test it, and what information is visible in HTTP traffic.

2. Learning Objectives

  1. Start a temporary HTTP service on Ubuntu-Target using Python.
  2. Explain the purpose of HTTP in client-server communication.
  3. Use curl to test HTTP connectivity from Kali-Lab.
  4. Use ss to verify whether a web service is listening locally.
  5. Use Nmap to verify remote visibility of TCP port 8080.
  6. Inspect HTTP status codes such as 200 OK and 404 Not Found.
  7. Inspect HTTP response headers and explain what information they reveal.
  8. Compare 127.0.0.1 binding with 0.0.0.0 binding.
  9. Optionally capture HTTP traffic in Wireshark and identify plaintext content.
  10. Document service testing evidence and OS-specific differences.

3. Lab Scenario

A small organization has asked a security analyst to verify whether a temporary internal web service is reachable from another system. The analyst must identify the service port, test the service, inspect HTTP behavior, review visible headers, and explain the security risk of leaving temporary services running.

6. Safety Rules

  1. Test only the assigned Ubuntu-Target VM.
  2. Do not scan or test public websites, school networks, home routers, cloud systems, or classmates’ machines.
  3. Keep both VMs on the isolated host-only/private network.
  4. Do not use bridged networking.
  5. Do not submit real passwords, private keys, personal browsing traffic, or unrelated captures.
  6. Do not leave temporary web services running after the lab.
  7. Use HTTP only for lab demonstration; do not transmit sensitive information over HTTP.

7. Key Concepts

HTTP

HTTP is an application-layer protocol used by web clients and web servers. In this lab, Kali acts as the HTTP client and Ubuntu acts as the HTTP server.

HTTP Request and Response

ItemMeaningExample
Request methodThe action requested by the clientGET / HTTP/1.1
Status codeThe server result code200 OK, 404 Not Found
HeaderMetadata about request or responseServer:, Content-Type:
BodyThe content returned by the serverHTML page text

Binding Address

BindingMeaningRemote Access?
127.0.0.1Listens only on local loopbackNo, usually only the same machine can access it
0.0.0.0Listens on all IPv4 interfacesYes, if network and firewall allow it
Specific lab IPListens only on that interfaceYes, on that interface if allowed

Plaintext Traffic

HTTP traffic is not encrypted. Headers and content may be visible in packet captures. This is why HTTPS is used for sensitive web communication.

Part 13 — Compare 127.0.0.1 Binding with 0.0.0.0 Binding

Stop the current Python web server on Ubuntu-Target using:

Ctrl + C

Start the web service bound only to loopback:

cd ~/Lab2-3-HTTP/web-root
python3 -m http.server 8080 --bind 127.0.0.1

In a second Ubuntu terminal, verify:

sudo ss -tulpen | grep ':8080'

From Ubuntu-Target, local access should work:

curl http://127.0.0.1:8080

From Kali-Lab, remote access should fail or time out:

curl --connect-timeout 5 -v http://UBUNTU-IP-ADDRESS:8080

Save the remote failure on Kali:

curl --connect-timeout 5 -v http://UBUNTU-IP-ADDRESS:8080 \
> ~/Lab2-3-HTTP/http-tests/loopback-binding-remote-failure.txt 2>&1 || true

Stop the loopback-bound service with Ctrl + C.

Required Screenshot 12: Loopback binding works locally on Ubuntu but fails remotely from Kali.

15. Troubleshooting Guide

ProblemLikely CauseFix
Kali cannot ping UbuntuWrong IP address or VMs are on different networksVerify ip -brief addr and use VMnet1 Host-only or Private to my Mac.
Python HTTP server will not startWrong directory, Python missing, or port already in useCheck python3 --version and sudo ss -tulpen | grep 8080.
curl returns connection refusedService not running or not listening on the expected interfaceStart the service and check binding with ss.
curl times outWrong IP, network issue, or service bound only to loopbackVerify ping and check whether service is bound to 127.0.0.1.
Nmap does not show port 8080 openService stopped, wrong target, wrong network, or binding issueConfirm with ss on Ubuntu and retry selected-port scan.
Wireshark does not show HTTPWrong interface or traffic not decoded as HTTPUse the correct interface and filter tcp.port == 8080.
404 test does not show 404Wrong path or cached/alternate responseUse a clearly missing file name such as /missing.html.
Port 8080 remains open after cleanupPython HTTP server still runningStop with Ctrl + C and verify with ss.
Evidence copy failsWrong username, IP, path, or SSH issueVerify SSH login and folder path.
Apple Silicon package issueWrong architecture image or packageUse ARM64 Kali/Ubuntu images and repositories.

16. Instructor Notes

Core lesson: HTTP service testing helps students connect application-layer behavior with ports, sockets, service exposure, and security risk.

Recommended Teaching Sequence

  1. Review TCP port 8080 and HTTP request/response basics.
  2. Create a simple web page on Ubuntu.
  3. Start Python HTTP server bound to 0.0.0.0.
  4. Test the service with curl from Kali.
  5. Inspect headers, verbose output, and 404 response.
  6. Verify port visibility with Nmap.
  7. Compare loopback binding versus all-interface binding.
  8. Discuss plaintext HTTP and exposed service information.
  9. Stop the temporary service and verify cleanup.

Recommended Short Version

  1. Start Python HTTP server on Ubuntu port 8080.
  2. Test with curl from Kali.
  3. Inspect headers with curl -I.
  4. Test a missing page to observe 404.
  5. Scan port 8080 with Nmap.
  6. Compare 127.0.0.1 and 0.0.0.0 binding.
  7. Stop service and submit report.
