1. Lab Purpose

The purpose of this lab is to help students understand password policy and offline hash risk without collecting or cracking real passwords. Students will create sample password policy rules, generate toy hashes from instructor-provided sample strings, compare salted and unsalted hash behavior, review Linux password-aging settings, and recommend a defensible password policy.

Main goal: Students should learn that password security depends on policy, user behavior, storage protections, salted password hashing, account controls, and monitoring—not just password complexity alone.

2. Learning Objectives

  1. Explain the purpose of password policies in access control.
  2. Differentiate password length, complexity, reuse prevention, expiration, and lockout concepts.
  3. Review Linux password policy files and password-aging settings safely.
  4. Explain why password hashes must be protected.
  5. Generate toy unsalted hashes using safe sample strings.
  6. Compare how small password changes produce different hash outputs.
  7. Compare unsalted and salted hash behavior using lab-only examples.
  8. Explain why offline hash exposure is dangerous.
  9. Create a password policy recommendation for a small organization.
  10. Explain OS-specific differences across Windows, Intel Mac, Apple Silicon Mac, Kali, and Ubuntu environments.

3. Lab Scenario

You are assisting a small organization with password policy improvement. The organization wants to understand why weak passwords, password reuse, poor storage, and exposed password hashes create risk. Your task is to review Linux password policy indicators, safely demonstrate hashing concepts using toy data, and write a policy recommendation.

  1. Do not collect, copy, submit, or display real password hashes from /etc/shadow.
  2. Do not attempt to crack real password hashes.
  3. Do not test passwords against real accounts, school systems, public services, or classmates’ accounts.
  4. Use only the toy sample strings provided in this lab.
  5. Do not submit actual passwords used by you or anyone else.
  6. Do not change password policies on production systems.
  7. Use only the assigned Ubuntu-Target and Kali-Lab VMs.
  8. Keep the lab network host-only/private.
Important: This lab teaches awareness of offline hash risk. It does not teach students to crack real password hashes. All hash examples are lab-created toy examples.

7. Key Concepts

Password Policy

A password policy defines requirements and protections for authentication secrets. Good policies balance usability and security while reducing weak, reused, or exposed passwords.

Password Hash

A password hash is a one-way transformation of a password. Systems should store password verifiers, not plaintext passwords. Modern password storage should use salted, slow password-hashing methods.

Salt

A salt is a unique random value added to a password before hashing. Salts help ensure that identical passwords do not produce identical stored hashes.

Offline Hash Risk

If password hashes are stolen, attackers may try guesses offline without interacting with the login system. This is why hash files, backups, logs, and databases containing password verifiers must be strongly protected.

Password Policy Categories

Policy AreaPurposeExample
LengthIncrease search space and reduce guessabilityMinimum 12–14 characters or longer passphrases
ComplexityEncourage variety, but should not replace lengthRequire a mix of character types only when appropriate
Reuse preventionPrevent repeated use of old passwordsPassword history controls
ExpirationLimit long-term exposure when justifiedChange after suspected compromise rather than arbitrary frequent changes
Lockout/rate limitingSlow online guessingAccount lockout or progressive delay
MFAReduce risk when a password is compromisedAuthenticator app, hardware token, push approval
StorageProtect password verifiersSalted slow hashes, restricted file/database permissions

cat ~/Lab3-3-PasswordPolicy/salted-hashes/salted-hash-comparison.txt

15. Troubleshooting Guide

ProblemLikely CauseFix
Kali cannot ping UbuntuWrong IP or VMs are on different virtual networksVerify ip -brief addr and host-only/private networking.
openssl command not foundOpenSSL is not installedUse Kali if OpenSSL is available there, or ask the instructor before installing packages.
chage requires permissionChecking another account requires sudoUse chage -l $(whoami) for your own account or sudo chage -l user for lab users.
PAM file output is confusingPAM configuration varies by distribution and versionDocument the lines present and explain them in general terms.
Hash output changes unexpectedlyInput includes a newline or different spacingUse printf instead of echo for controlled hash input.
Salted hash output is longCrypt-style hashes include algorithm, salt, and hash valueBreak the output into parts: algorithm marker, salt, hash output.
Student accidentally displays /etc/shadowIncorrect command usedDo not submit the output. Delete screenshots containing real hashes and notify the instructor.
scp failsWrong username, IP, path, or SSH issueVerify SSH access and evidence folder path.
