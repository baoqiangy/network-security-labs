Lab 8.3 — Endpoint Hardening Checklist

Network Security hands-on lab based on the Lab 1.1 virtual environment. Students perform a structured endpoint hardening review on Ubuntu and compare relevant Windows/macOS hardening concepts. The lab covers patch status, account review, password and sudo policy, SSH hardening, firewall rules, unnecessary services, file permissions, logging, audit readiness, and final hardening recommendations.

1. Lab Purpose

The purpose of this lab is to help students learn how to assess and improve endpoint security using a practical hardening checklist. Students will review the current configuration of an Ubuntu endpoint, identify weaknesses, apply safe hardening steps where assigned, and document evidence-based recommendations.

Main goal: Students should understand that endpoint hardening is a process of reducing attack surface, enforcing least privilege, improving authentication controls, enabling logging, and verifying configuration.

2. Learning Objectives

  1. Explain the purpose of endpoint hardening in network security.
  2. Collect baseline endpoint configuration evidence.
  3. Review operating system version, patch status, kernel information, and installed packages.
  4. Identify local users, administrative privileges, stale accounts, and risky sudo settings.
  5. Review password policy and authentication-related settings.
  6. Evaluate SSH configuration for common security weaknesses.
  7. Review host firewall status and create least-privilege firewall rules where assigned.
  8. Identify unnecessary services and listening ports.
  9. Review file permissions on selected sensitive files and directories.
  10. Verify logging and audit readiness.
  11. Create a prioritized hardening checklist and risk-based recommendation report.
  12. Compare endpoint hardening differences across Ubuntu, Kali, Windows, macOS, and cloud endpoints.

3. Lab Scenario

You are a security analyst asked to review a Linux endpoint before it is used in a small organization’s internal network. The system was installed quickly, and management wants assurance that basic hardening has been reviewed. Your job is to collect evidence, assess common security controls, apply safe changes if assigned, and create a final endpoint hardening checklist.

6. Safety Rules

  1. Use only the assigned lab VM or instructor-provided endpoint.
  2. Do not apply hardening changes to production systems.
  3. Do not disable your only administrator account.
  4. Do not lock yourself out of SSH unless the instructor explicitly requires a local console test.
  5. Back up configuration files before editing them.
  6. Do not delete system users, packages, or services unless instructed.
  7. Do not expose the endpoint to public networks.
  8. Do not include passwords, private keys, tokens, or sensitive personal data in submissions.
Important: Some hardening changes can break access or services. In this lab, students should review first, back up configuration files, then make only safe instructor-approved changes.

7. Key Concepts

Endpoint Hardening

Endpoint hardening is the process of reducing the security risk of a workstation, server, or virtual machine by limiting attack surface, strengthening authentication, removing unnecessary services, applying patches, enabling logging, and enforcing least privilege.

Attack Surface

Attack surface includes all the ways an endpoint can be accessed or attacked, such as users, services, ports, applications, credentials, and exposed files.

Least Privilege

Least privilege means users and services should have only the access they need. For endpoints, this includes limiting sudo/admin rights, service permissions, and network exposure.

Secure Baseline

A secure baseline is a documented standard configuration. Hardening compares the current system against a baseline and identifies deviations.

Compensating Control

A compensating control is an alternative security measure used when a preferred control cannot be fully implemented.

11. OS-Specific Differences

Ubuntu Endpoint

Ubuntu is the primary endpoint for this lab. Hardening checks focus on apt, ufw, systemctl, journalctl, sshd_config, sudo, and Linux file permissions.

sudo apt update
apt list --upgradable
sudo ufw status verbose
sudo ss -tulpen
sudo systemctl --type=service --state=running
sudo journalctl --since "30 minutes ago" --no-pager

Kali Linux

Kali is used as an assessment workstation. It can test network reachability and exposed ports on Ubuntu-Target.

ping -c 4 UBUNTU-IP
ssh USER@UBUNTU-IP "hostname && whoami"
nmap -sT -Pn -p 22,80,443,8080 UBUNTU-IP
scp -r USER@UBUNTU-IP:/home/USER/Lab8-3-EndpointHardening ./ubuntu-evidence
