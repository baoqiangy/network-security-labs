1. Lab Purpose

The purpose of this lab is to introduce packet capture and protocol analysis using Wireshark. Students will generate safe traffic in the isolated lab environment and analyze what happens at different layers of the network stack.

Main goal: Students should understand how packets reveal network behavior, including address resolution, reachability testing, name resolution, connection setup, and web traffic.

2. Learning Objectives

  1. Launch Wireshark and select the correct capture interface.
  2. Capture ARP traffic between Kali-Lab and Ubuntu-Target.
  3. Capture ICMP traffic using ping.
  4. Capture DNS traffic using a safe lab or temporary NAT option.
  5. Capture TCP three-way handshake traffic.
  6. Capture HTTP request and response traffic from a temporary local web service.
  7. Use Wireshark display filters to isolate protocols.
  8. Identify source IP, destination IP, protocol, ports, TCP flags, and basic payload details.
  9. Save capture files and annotated evidence screenshots.
  10. Explain OS-specific differences across Windows, Intel Mac, Apple Silicon Mac, Kali, and Ubuntu environments.

3. Lab Scenario

You are a network security analyst learning to observe normal traffic before studying attacks, alerts, and incident response. Your instructor has asked you to capture common protocol traffic in a controlled lab so you can recognize ARP, ICMP, DNS, TCP, and HTTP packets in Wireshark.

6. Safety Rules

  1. Capture traffic only inside your assigned lab environment.
  2. Do not capture traffic from public Wi-Fi, school production networks, home networks, or classmates’ devices.
  3. Do not capture passwords, private information, or personal browsing traffic.
  4. Do not use bridged networking unless specifically authorized by the instructor.
  5. Do not run denial-of-service tools or attack traffic.
  6. Use the temporary HTTP service only on Ubuntu-Target and stop it after the lab.
  7. If temporary Internet access is used for DNS capture, switch back to host-only/private networking afterward.
Important: Packet capture can expose sensitive information. Keep this lab limited to assigned VMs and lab-generated traffic only.

7. Key Concepts

Packet Capture

Packet capture records network packets seen on a network interface. Wireshark displays these packets so analysts can inspect protocol behavior, addresses, ports, flags, and payload details.

Display Filter vs. Capture Filter

Filter TypePurposeExample
Capture filterLimits what is captured before packets are savedhost 192.168.56.102
Display filterFilters what is displayed after captureicmp, arp, tcp.port == 8080

Protocols Captured in This Lab

ProtocolPurposeDisplay Filter
ARPMaps IP addresses to MAC addresses on the local networkarp
ICMPUsed by ping to test reachabilityicmp
DNSResolves names to IP addressesdns
TCPConnection-oriented transport protocoltcp
HTTPUnencrypted web traffichttp or tcp.port == 8080

Part 5 — Start a Full Lab Capture

  1. In Wireshark, select the correct host-only/private interface.
  2. Click the blue shark fin icon to start capturing.
  3. Do not apply a capture filter for the first run. You will use display filters after traffic is captured.

Required Screenshot 5: Wireshark actively capturing packets.

Part 6 — Capture ARP Traffic

On Kali-Lab:

ip neigh
ping -c 2 UBUNTU-IP-ADDRESS

In Wireshark, apply:

arp

Look for “Who has UBUNTU-IP?” and the matching ARP reply. Save the capture as arp-capture.pcapng.

Required Screenshot 6: ARP packets displayed with the arp filter.

Part 7 — Capture ICMP Traffic

On Kali-Lab:

ping -c 4 UBUNTU-IP-ADDRESS

In Wireshark, apply:

icmp

Identify ICMP Echo Request and Echo Reply packets. Save the capture as icmp-capture.pcapng.

Required Screenshot 7: ICMP Echo Request and Echo Reply packets.

Part 8 — Prepare a Temporary HTTP Service on Ubuntu

On Ubuntu-Target:

cd ~/Lab2-1-Wireshark/http-service
cat > index.html <<'EOF'


Lab 2.1 HTTP Capture

  
Lab 2.1 Packet Capture with Wireshark

  
This page is served by Ubuntu-Target for a local HTTP capture.



EOF

python3 -m http.server 8080 --bind 0.0.0.0

Leave the server running. In another Ubuntu terminal:

sudo ss -tulpen | grep ':8080'

Required Screenshot 8: Temporary HTTP service listening on TCP/8080.

Part 9 — Capture TCP Three-Way Handshake

From Kali-Lab:

curl -I http://UBUNTU-IP-ADDRESS:8080

In Wireshark, apply:

tcp.port == 8080

Find the TCP handshake:

StepPacketMeaning
1SYNKali requests a TCP connection
2SYN, ACKUbuntu acknowledges and agrees
3ACKKali completes the handshake

Save the capture as tcp-handshake-capture.pcapng.

Required Screenshot 9: TCP three-way handshake packets for port 8080.

Part 10 — Capture HTTP Request and Response

From Kali-Lab:

curl http://UBUNTU-IP-ADDRESS:8080

In Wireshark, use:

http

or:

tcp.port == 8080

Identify the HTTP GET request, HTTP response status, IP addresses, TCP ports, and visible HTTP content. Save as http-capture.pcapng.

Required Screenshot 10: HTTP GET and response traffic.

Part 11 — Capture DNS Traffic

DNS in isolation: In a fully isolated host-only network, external DNS may fail. That is acceptable if the instructor wants the lab fully isolated. Document the result.

Option A — Temporary NAT or Share with My Mac

Use only if the instructor allows temporary Internet access. Temporarily switch Kali-Lab to NAT on Windows VMware Workstation Pro or Share with my Mac on VMware Fusion. Then run:

dig example.com
# or
nslookup example.com

In Wireshark, apply:

dns

Save as dns-capture.pcapng, then return Kali-Lab to host-only/private networking.

Option B — Instructor-Provided Lab DNS Server

dig @LAB-DNS-IP lab.example

Filter with dns and save the capture.

Option C — DNS Unavailable Due to Isolation

If DNS fails because the network is isolated, document it:

dig example.com > ~/Lab2-1-Wireshark/dns/dns-unavailable-isolated-network.txt 2>&1 || true

Required Screenshot 11: DNS query/response, or DNS unavailable due to isolation.

15. Troubleshooting Guide

ProblemLikely CauseFix
No packets appear in WiresharkWrong interface selectedUse ip -brief addr to identify the interface with the Kali lab IP.
Kali cannot ping UbuntuWrong IP or network mismatchVerify both VMs use host-only/private networking and check IP addresses.
No ARP packets appearARP cache already has an entryPing again or document existing neighbor information using ip neigh.
No HTTP packets appearHTTP service not running or wrong portCheck the Python HTTP server and ss -tulpen | grep ':8080'.
Wireshark does not show HTTP filter resultsTraffic may not decode as HTTP automaticallyUse tcp.port == 8080 instead.
DNS query failsLab network is isolatedDocument the isolation result or use instructor-approved DNS option.
Wireshark permission issueUser lacks capture permissionRun sudo wireshark in the lab VM.
Port 8080 remains open after cleanupPython web server is still runningPress Ctrl + C in the server terminal and verify with ss.
