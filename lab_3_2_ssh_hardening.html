Lab 3.2 — SSH Hardening

Network Security hands-on lab based on the Lab 1.1 virtual environment. Students harden SSH on Ubuntu-Target by documenting the baseline, creating SSH keys, enabling key-based login, disabling root login, testing password-authentication behavior, reviewing firewall access, and preparing a rollback plan.

1. Lab Purpose

The purpose of this lab is to teach students how to harden SSH access safely. Students will first document the current SSH configuration, then create and test SSH key authentication before applying hardening controls. Students will verify each change from Kali-Lab so they do not rely only on local configuration files.

Main goal: Students should understand that SSH hardening must be performed in a safe order: document baseline, configure keys, test keys, validate configuration syntax, reload SSH, test from another host, and keep a rollback path.

2. Learning Objectives

  1. Explain the security purpose of SSH hardening.
  2. Identify the SSH service status, listening port, and configuration files.
  3. Generate an SSH key pair on Kali-Lab.
  4. Install the public key on Ubuntu-Target.
  5. Test key-based SSH login from Kali-Lab to Ubuntu-Target.
  6. Disable root login for SSH.
  7. Optionally disable password authentication after key-based login is confirmed.
  8. Validate SSH configuration syntax before reloading the service.
  9. Verify SSH hardening with ssh, nmap, ss, systemctl, and logs.
  10. Document OS-specific differences across Windows, Intel Mac, Apple Silicon Mac, Kali, and Ubuntu environments.

3. Lab Scenario

A small organization uses SSH to administer an Ubuntu server. The security team wants to reduce risk by preventing root login, using SSH keys, limiting unnecessary authentication methods, and confirming that firewall rules allow only required administrative access. You are responsible for hardening SSH while preserving authorized access.

6. Safety Rules

  1. Harden SSH only on your assigned Ubuntu-Target VM.
  2. Do not change SSH settings on host machines, school systems, cloud servers, or production systems.
  3. Keep an open console session to Ubuntu-Target while changing SSH configuration.
  4. Do not disable password authentication until key-based login has been tested successfully.
  5. Always run sudo /usr/sbin/sshd -t before reloading SSH.
  6. Do not share or submit private SSH keys.
  7. Submit only screenshots, public-key evidence if requested, configuration summaries, and test outputs.
  8. Keep the lab network host-only/private.
Important: SSH misconfiguration can lock you out. This lab uses a drop-in config file and requires syntax testing before reload.

7. Key Concepts

SSH

Secure Shell, or SSH, provides encrypted remote access to a system. It is commonly used for Linux server administration and must be protected carefully.

Password Authentication vs. Key-Based Authentication

MethodDescriptionSecurity Consideration
Password authenticationUser logs in with a passwordCan be vulnerable to guessing, reuse, and brute-force attempts
Key-based authenticationUser logs in using a private key matched to a public key on the serverStronger when private keys are protected and passphrases are used

Important SSH Hardening Settings

SettingRecommended Lab ValuePurpose
PermitRootLoginnoPrevents direct root login over SSH
PubkeyAuthenticationyesAllows SSH key authentication
PasswordAuthenticationno after keys are confirmedPrevents password-based login attempts
MaxAuthTries3Limits repeated failed authentication attempts
AllowUsersAuthorized lab username onlyRestricts which users may log in through SSH

Drop-In Configuration Files

Many modern Linux systems support SSH server drop-in files under /etc/ssh/sshd_config.d/. This lab uses a file named 99-lab-ssh-hardening.conf so students do not need to edit the main /etc/ssh/sshd_config file directly.

Part 10 — Create a Lab SSH Hardening Drop-In File

On Ubuntu-Target, replace YOUR-UBUNTU-USERNAME with your actual lab username:

sudo tee /etc/ssh/sshd_config.d/99-lab-ssh-hardening.conf > /dev/null <<'EOF'
# Lab 3.2 SSH hardening drop-in configuration
PermitRootLogin no
PubkeyAuthentication yes
PasswordAuthentication no
KbdInteractiveAuthentication no
MaxAuthTries 3
AllowUsers YOUR-UBUNTU-USERNAME
EOF

Review the file:

sudo cat /etc/ssh/sshd_config.d/99-lab-ssh-hardening.conf

Save evidence:

sudo cat /etc/ssh/sshd_config.d/99-lab-ssh-hardening.conf > ~/Lab3-2-SSHHardening/ssh-config/99-lab-ssh-hardening.conf.txt
Important: The AllowUsers line must contain your real Ubuntu username. If it contains the placeholder text, SSH login may fail.

11. OS-Specific Differences

Kali-Lab

Kali is the SSH client and testing workstation. Keys are generated on Kali, and SSH access is tested from Kali to Ubuntu.

ssh-keygen -t ed25519 -f ~/.ssh/lab32_ed25519
ssh-copy-id -i ~/.ssh/lab32_ed25519.pub user@UBUNTU-IP
ssh -i ~/.ssh/lab32_ed25519 user@UBUNTU-IP
nmap -sV -p 22 UBUNTU-IP

Ubuntu-Target

Ubuntu is the SSH server and hardening target. SSH server configuration is performed on Ubuntu.

sudo systemctl status ssh --no-pager
sudo ss -tulpen | grep ':22'
sudo /usr/sbin/sshd -t
sudo /usr/sbin/sshd -T
sudo systemctl reload ssh
sudo journalctl -u ssh --no-pager

15. Troubleshooting Guide
