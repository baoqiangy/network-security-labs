Lab 4.1 — Hashing and File Integrity

Network Security hands-on lab based on the Lab 1.1 virtual environment. Students generate cryptographic hashes, verify file integrity, detect file tampering, compare hash algorithms, and document integrity evidence using Kali-Lab and Ubuntu-Target.

1. Lab Purpose

This lab introduces cryptographic hashing as a file-integrity tool. Students create test files, calculate hashes, transfer files between VMs, detect file changes, compare algorithm output, and write a short integrity verification report.

Kali-Lab ───── Host-only / Private Network ───── Ubuntu-Target
Main goal: Students should understand that a small change in file content produces a different hash, making hashing useful for detecting tampering or verifying file integrity.

2. Learning Objectives

  1. Explain the role of hashing in file integrity and digital evidence handling.
  2. Generate MD5, SHA-1, and SHA-256 hashes for files.
  3. Explain why SHA-256 is preferred over older algorithms for integrity verification.
  4. Create a baseline hash record for a file.
  5. Transfer a file between Kali-Lab and Ubuntu-Target using scp.
  6. Verify whether a transferred file matches the original file.
  7. Modify a file and detect the change using hash comparison.
  8. Use sha256sum -c to automate integrity checking.
  9. Document hash values, evidence files, and integrity conclusions.
  10. Explain OS-specific differences across Windows, Intel Mac, Apple Silicon Mac, Kali, and Ubuntu.

3. Lab Scenario

A security analyst receives a configuration file from a server and needs to verify whether the file remains unchanged after transfer and later handling. The analyst creates a known-good hash, copies the file between systems, verifies it, then intentionally modifies a copy to observe tamper detection.

6. Safety Rules

  1. Use only files created for this lab.
  2. Do not hash, copy, or submit files containing passwords, private keys, personal data, or sensitive documents.
  3. Do not modify operating system files.
  4. Do not use bridged networking.
  5. Keep both VMs on the isolated host-only/private network.
  6. Do not claim a file was tampered with unless the hash comparison supports that conclusion.

7. Key Concepts

Hash Function

A hash function produces a fixed-length value from input data. For file integrity, the hash value acts like a fingerprint of the file content.

Common Hash Algorithms

AlgorithmTypical Output LengthCurrent Security Use
MD5128-bit hashNot recommended for security; useful only for basic non-security checks or legacy comparison
SHA-1160-bit hashNot recommended for modern security uses
SHA-256256-bit hashRecommended for file integrity verification in this lab

Part 5 — Generate Baseline Hashes on Ubuntu

cd ~/Lab4-1-Hashing/source-files
md5sum system_config.txt
sha1sum system_config.txt
sha256sum system_config.txt
md5sum system_config.txt > ~/Lab4-1-Hashing/hashes/system_config.md5
sha1sum system_config.txt > ~/Lab4-1-Hashing/hashes/system_config.sha1
sha256sum system_config.txt > ~/Lab4-1-Hashing/hashes/system_config.sha256
{
  echo "MD5:"; md5sum system_config.txt
  echo; echo "SHA-1:"; sha1sum system_config.txt
  echo; echo "SHA-256:"; sha256sum system_config.txt
} > ~/Lab4-1-Hashing/hashes/system_config_all_hashes.txt

Part 6 — Generate an OpenSSL SHA-256 Hash

openssl dgst -sha256 ~/Lab4-1-Hashing/source-files/system_config.txt
openssl dgst -sha256 ~/Lab4-1-Hashing/source-files/system_config.txt \
> ~/Lab4-1-Hashing/hashes/system_config_openssl_sha256.txt

Compare the OpenSSL SHA-256 value with the sha256sum value. The hash value should match even though the output format is different.

Part 8 — Verify File Integrity on Kali

cp ~/Lab4-1-Hashing/hashes/system_config.sha256 ~/Lab4-1-Hashing/received-files/

sha256sum -c system_config.sha256 > ~/Lab4-1-Hashing/comparisons/sha256-verification-original.txt

Expected result: system_config.txt: OK.

Part 9 — Modify the Received File on Kali

cd ~/Lab4-1-Hashing/received-files
cp system_config.txt system_config_modified.txt
echo "Unauthorized Change: Enabled" >> system_config_modified.txt
cat system_config_modified.txt
sha256sum system_config_modified.txt > ~/Lab4-1-Hashing/hashes/system_config_modified.sha256
cat ~/Lab4-1-Hashing/hashes/system_config_modified.sha256

Part 10 — Compare Original and Modified Hashes

cat ~/Lab4-1-Hashing/hashes/system_config.sha256
cat ~/Lab4-1-Hashing/hashes/system_config_modified.sha256
{
  echo "Original file SHA-256:"; cat ~/Lab4-1-Hashing/hashes/system_config.sha256
  echo; echo "Modified file SHA-256:"; cat ~/Lab4-1-Hashing/hashes/system_config_modified.sha256
} > ~/Lab4-1-Hashing/comparisons/original-vs-modified-sha256.txt

Part 11 — Test a Failed Integrity Check

cd ~/Lab4-1-Hashing/received-files
ORIGINAL_HASH=$(cut -d ' ' -f1 system_config.sha256)
echo "$ORIGINAL_HASH  system_config_modified.txt" > check_modified_against_original.sha256
sha256sum -c check_modified_against_original.sha256
sha256sum -c check_modified_against_original.sha256 \
> ~/Lab4-1-Hashing/comparisons/sha256-verification-modified-failed.txt 2>&1 || true

Expected result: system_config_modified.txt: FAILED.

Part 12 — Use diff to Identify the Content Change

diff -u system_config.txt system_config_modified.txt \
> ~/Lab4-1-Hashing/comparisons/file-content-difference.txt || true

15. Troubleshooting Guide

ProblemLikely CauseFix
Kali cannot ping UbuntuWrong IP or VM network mismatchVerify ip -brief addr and host-only/private network mode.

16. Instructor Notes

Core lesson: Hashing verifies integrity but does not automatically prove who made a change, why it changed, or whether the original file was trustworthy. Students should separate evidence from interpretation.

Recommended Teaching Sequence

  1. Review the purpose of hashes in integrity verification.
  2. Create a baseline file on Ubuntu.
  3. Generate MD5, SHA-1, SHA-256, and OpenSSL hashes.
  4. Copy the file and baseline hash to Kali.
  5. Verify the copied file with sha256sum -c.
  6. Modify a copy and show failed verification.
  7. Use diff to connect hash differences to content differences.
  8. Discuss why SHA-256 is preferred.
  9. Complete the integrity report.
