1. Lab Purpose

This lab introduces packet capture and packet analysis using Wireshark. Students will capture traffic generated between Kali-Lab and Ubuntu-Target, identify common protocols, apply display filters, save packet-capture files, and explain what they observed.

Kali-Lab ───── Host-only / Private Network ───── Ubuntu-Target
Main goal: Learn how to capture and interpret basic traffic safely inside an isolated virtual lab network.

2. Learning Objectives

  1. Explain the purpose of packet capture in network security.
  2. Identify the correct capture interface in a virtual network environment.
  3. Use Wireshark to capture ICMP, ARP, HTTP, and SSH-related traffic.
  4. Use display filters to focus on specific protocols, ports, or IP addresses.
  5. Compare Wireshark packet captures with command-line network tests.
  6. Save packet captures as .pcapng files.
  7. Document evidence using screenshots and written observations.
  8. Explain differences across Windows, Intel Mac, Apple Silicon Mac, Kali, and Ubuntu environments.

3. Lab Scenario

A security analyst needs to verify what traffic is moving between two internal lab systems. The analyst uses Wireshark to capture traffic and answer the following questions:

  1. Which IP addresses are communicating?
  2. Which protocols are visible?
  3. What does a ping look like in Wireshark?
  4. What does SSH traffic look like compared with HTTP traffic?
  5. How can display filters help analyze a packet capture?

6. Safety Rules

  1. Capture traffic only inside the assigned Kali-Lab and Ubuntu-Target virtual network.
  2. Do not capture traffic on public, school, work, or home networks.
  3. Do not use bridged networking.
  4. Do not capture or submit real passwords, private keys, personal messages, or unrelated network traffic.
  5. Do not scan or test systems outside the lab environment.
  6. Keep packet capture files limited to the traffic generated in this lab.
  7. Use screenshots that show packet details but do not expose sensitive personal information.

7. Key Concepts

Packet Capture

A packet capture records network packets seen on a network interface. Analysts use packet captures to troubleshoot, validate, and investigate network activity.

Capture Filter vs. Display Filter

Filter TypePurposeExample
Capture filterLimits what is captured before capture beginshost 192.168.56.102
Display filterFilters what is shown after packets are capturedip.addr == 192.168.56.102

Common Protocols in This Lab

ProtocolWhat Students Should Observe
ARPAddress resolution between IP and MAC addresses
ICMPPing echo request and echo reply packets
SSHEncrypted TCP session, usually on port 22
HTTPPlaintext web request and response on port 8080 in this lab
TCPConnection setup and data transport

9. Detailed Lab Steps

Part 1 — Verify the Lab Network

Start both VMs. On Ubuntu-Target, find the IP address:

hostname -I

On Kali-Lab, find the IP address:

ip addr
ip route
uname -m

From Kali-Lab, test connectivity to Ubuntu:

ping -c 4 UBUNTU-IP-ADDRESS

Required Screenshot 1: Successful ping from Kali-Lab to Ubuntu-Target.

Part 2 — Verify or Install Wireshark

On Kali-Lab, check Wireshark:

wireshark --version

If Wireshark is missing, temporarily use NAT or Share with my Mac for Internet access, then install:

sudo apt update
sudo apt install -y wireshark tcpdump

If prompted whether non-superusers should be able to capture packets, select the instructor-approved option. If unsure, choose No and run Wireshark with sudo for this lab.

sudo wireshark
After installing packages: Return the VM network setting to VMnet1 Host-only on Windows or Private to my Mac on macOS before continuing.

Required Screenshot 2: Wireshark installed or available on Kali.

Part 3 — Create Lab Evidence Folders

On Kali-Lab:

mkdir -p ~/Lab1-2-Wireshark/{captures,network-info,screenshots,notes,report}

On Ubuntu-Target:

mkdir -p ~/Lab1-2-Wireshark/{network-info,web-test,notes}

Required Screenshot 3: Evidence folders created on both VMs.

Part 4 — Save Network Information

On Kali-Lab:

hostname > ~/Lab1-2-Wireshark/network-info/kali-hostname.txt
ip addr > ~/Lab1-2-Wireshark/network-info/kali-ip-addr.txt

Required Screenshot 4: Saved network information files.

Part 5 — Identify the Correct Capture Interface

On Kali-Lab, open Wireshark:

sudo wireshark

Look at the list of available interfaces. The correct interface is usually the one with traffic activity and an IP address on the same network as Ubuntu.

Common Linux interface names include:

eth0
ens33
enp0s3

Confirm the interface in a terminal:

ip addr

Required Screenshot 5: Wireshark interface list and the selected capture interface.

Part 6 — Capture ICMP Ping Traffic

  1. In Wireshark on Kali, double-click the correct interface to start capturing.
  2. In a Kali terminal, ping Ubuntu:
ping -c 4 UBUNTU-IP-ADDRESS
  1. Stop the capture in Wireshark.
  2. Apply this display filter:
icmp

Students should observe echo request and echo reply packets.

Save the capture:

File → Save As → ~/Lab1-2-Wireshark/captures/icmp-ping-capture.pcapng

Required Screenshot 6: ICMP packets shown in Wireshark using the icmp filter.

Part 7 — Capture ARP Traffic

Start a new capture in Wireshark. Generate traffic by pinging Ubuntu again:

ping -c 4 UBUNTU-IP-ADDRESS

Apply the ARP display filter:

arp

If no ARP packets appear, clear the ARP cache and try again.

On Kali:

sudo ip neigh flush all
ping -c 4 UBUNTU-IP-ADDRESS

Save the capture:

File → Save As → ~/Lab1-2-Wireshark/captures/arp-capture.pcapng

Required Screenshot 7: ARP request or ARP reply packets in Wireshark.

Part 8 — Capture SSH Traffic

Start a new capture in Wireshark on Kali. In a Kali terminal, connect to Ubuntu with SSH:

ssh YOUR-UBUNTU-USERNAME@UBUNTU-IP-ADDRESS

After logging in:

whoami
hostname
exit

Stop the Wireshark capture and apply one of these filters:

tcp.port == 22

or:

ip.addr == UBUNTU-IP-ADDRESS && tcp.port == 22

Save the capture:

File → Save As → ~/Lab1-2-Wireshark/captures/ssh-capture.pcapng
Observation: SSH traffic should be visible as encrypted TCP traffic. Students should not expect to read the username or command output in plaintext inside the packet payload.

Required Screenshot 8: SSH packets filtered by TCP port 22.

Part 9 — Start a Temporary Web Service on Ubuntu

On Ubuntu-Target, open a new terminal:

mkdir -p ~/Lab1-2-Wireshark/web-test
cd ~/Lab1-2-Wireshark/web-test
echo "Lab 1.2 Wireshark HTTP test page" > index.html
python3 -m http.server 8080 --bind 0.0.0.0

Leave this terminal running. In another Ubuntu terminal, verify the service:

sudo ss -tulpen | grep 8080

Required Screenshot 9: Temporary web server running on Ubuntu port 8080.

Part 10 — Capture HTTP Traffic

Start a new Wireshark capture on Kali. In a Kali terminal, request the web page:

curl http://UBUNTU-IP-ADDRESS:8080

Expected output:

Lab 1.2 Wireshark HTTP test page

Stop the capture. Apply one of these display filters:

tcp.port == 8080

or:

http

Save the capture:

File → Save As → ~/Lab1-2-Wireshark/captures/http-8080-capture.pcapng
Important: HTTP traffic may show readable request and response data. This is one reason sensitive traffic should use encryption in real networks.

Required Screenshot 10: HTTP or TCP port 8080 traffic in Wireshark.

Part 11 — Use IP-Based Display Filters

Open one of the saved captures and apply these filters:

ip.addr == UBUNTU-IP-ADDRESS
ip.src == KALI-IP-ADDRESS
ip.dst == UBUNTU-IP-ADDRESS

Students should observe how filters narrow the packet list.

Required Screenshot 11: IP-based display filter showing only Kali-to-Ubuntu or Ubuntu-related traffic.

Part 12 — Optional: Compare Wireshark with tcpdump

On Kali-Lab, identify the interface:

ip addr

Run tcpdump for ICMP traffic:

sudo tcpdump -i INTERFACE-NAME icmp

Example:

sudo tcpdump -i eth0 icmp

In another Kali terminal, ping Ubuntu:

ping -c 4 UBUNTU-IP-ADDRESS

Stop tcpdump with Ctrl + C.

16. Instructor Notes

Core lesson: Packet capture is one of the most important evidence-gathering skills in network security. Students should learn not only how to capture packets, but also how to explain what the packets prove.

Recommended Teaching Sequence

  1. Review the Lab 1.1 topology.
  2. Explain capture interfaces and display filters.
  3. Capture ICMP traffic first because it is easy to recognize.
  4. Capture ARP after flushing the neighbor table.
  5. Capture SSH and discuss encryption.
  6. Capture HTTP and discuss plaintext exposure.
  7. Have students save captures and complete observations.
  8. End with a short report connecting packet capture to network security monitoring.

Recommended Short Version

  1. Open Wireshark on Kali.
  2. Capture ping traffic.
  3. Capture SSH traffic.
  4. Start a Python web server on Ubuntu.
  5. Capture HTTP traffic.
  6. Apply display filters and save captures.
  7. Complete the observation table and report.
