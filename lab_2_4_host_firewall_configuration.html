1. Lab Purpose

The purpose of this lab is to help students configure and verify a host-based firewall. Students will observe baseline exposure, enable UFW on Ubuntu-Target, allow SSH, block or deny a temporary HTTP service, then verify results using Kali-Lab.

Main goal: Students should understand that firewall rules must be tested from another system. A rule is not meaningful unless the defender can verify what is allowed and what is blocked.

2. Learning Objectives

  1. Explain the role of a host-based firewall in network security.
  2. View baseline firewall status and listening services on Ubuntu-Target.
  3. Enable UFW safely without locking out SSH.
  4. Configure default deny inbound and allow outbound policy.
  5. Allow SSH access from Kali-Lab.
  6. Start a temporary HTTP service and test its reachability.
  7. Block or deny HTTP service access using UFW.
  8. Verify firewall behavior using nmap, curl, and ssh.
  9. Document firewall rules, test results, and security meaning.
  10. Explain OS-specific differences across Windows, Intel Mac, Apple Silicon Mac, Kali, and Ubuntu environments.

3. Lab Scenario

You are hardening a small Ubuntu server in a private lab network. SSH should remain available for administration, but unnecessary web access should be blocked. Your job is to configure host firewall rules and prove that the firewall allows required access while blocking unnecessary exposure.

6. Safety Rules

  1. Configure firewall rules only on your assigned Ubuntu-Target VM.
  2. Do not scan or test public systems, school networks, home routers, or classmates’ devices.
  3. Do not disable firewall protections on real systems.
  4. Before enabling UFW, allow SSH so you do not accidentally block remote administration.
  5. Use the temporary HTTP service only inside the isolated lab network.
  6. Stop the temporary HTTP service before finishing the lab.
  7. Keep the network host-only/private unless the instructor explicitly authorizes otherwise.
Important: Firewall configuration can lock you out of a system if done carelessly. Follow the rule order exactly and keep console access to Ubuntu-Target available.

7. Key Concepts

Host-Based Firewall

A host-based firewall filters traffic entering or leaving a single computer. In this lab, Ubuntu-Target uses UFW to control inbound access to services.

Default Deny

A default deny inbound policy blocks unsolicited inbound traffic unless a specific allow rule exists. This is a common defensive posture for servers and endpoints.

Allow Rule vs. Deny Rule

Rule TypeMeaningExample
AllowPermits matching trafficsudo ufw allow from KALI-IP to any port 22 proto tcp
DenyBlocks matching trafficsudo ufw deny 8080/tcp
Default deny inboundBlocks inbound traffic that does not match an allow rulesudo ufw default deny incoming

Testing Firewall Behavior

TestToolWhat It Shows
Is SSH reachable?ssh, nmap -p 22Whether administrative access is allowed
Is web service reachable?curl, nmap -p 8080Whether the HTTP service is exposed
What is listening locally?ss -tulpenWhether a service is listening even if blocked remotely
What rules exist?ufw status numberedFirewall policy and rule order

Part 6 — Safely Configure Default Firewall Policy

Rule order matters: Add an SSH allow rule before enabling UFW. This helps prevent accidental lockout when using SSH.

On Ubuntu-Target, replace KALI-IP-ADDRESS with the actual Kali IP:

sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow from KALI-IP-ADDRESS to any port 22 proto tcp
sudo ufw --force enable
sudo ufw status verbose
sudo ufw status numbered

Save evidence:

sudo ufw status verbose > ~/Lab2-4-Firewall/firewall-rules/ufw-status-after-enable.txt
sudo ufw status numbered > ~/Lab2-4-Firewall/firewall-rules/ufw-status-numbered-after-enable.txt

Required Screenshot 6: UFW enabled with default deny incoming, allow outgoing, and SSH allowed from Kali.

Part 11 — Remove HTTP 8080 Allow Rule and Verify Block Again

On Ubuntu-Target, list numbered rules:

sudo ufw status numbered

Find the rule allowing port 8080 from Kali and delete it. Replace RULE-NUMBER with the correct number:

sudo ufw delete RULE-NUMBER
sudo ufw status numbered

If you prefer, you may use the equivalent rule deletion form:

sudo ufw delete allow from KALI-IP-ADDRESS to any port 8080 proto tcp

Save evidence:

sudo ufw status numbered > ~/Lab2-4-Firewall/firewall-rules/ufw-status-after-delete-8080.txt

On Kali-Lab, verify HTTP is blocked again:

curl -I --connect-timeout 5 http://UBUNTU-IP-ADDRESS:8080 > ~/Lab2-4-Firewall/block-http-tests/http-8080-after-delete-curl.txt 2>&1 || true

nmap -sT -p 8080 UBUNTU-IP-ADDRESS -oA ~/Lab2-4-Firewall/block-http-tests/ubuntu-8080-after-delete-scan

16. Instructor Notes

Core lesson: Firewall rules should be configured with least privilege and verified from the network. Students should distinguish between a service listening locally and a service being reachable remotely.

Recommended Teaching Sequence

  1. Review host firewall concepts and rule order.
  2. Have students document the baseline state before enabling UFW.
  3. Allow SSH from Kali before enabling UFW.
  4. Show that default deny blocks a temporary service even when it is locally listening.
  5. Temporarily allow and then remove the 8080 rule to demonstrate rule behavior.
  6. Discuss firewall documentation, change management, and least privilege.
