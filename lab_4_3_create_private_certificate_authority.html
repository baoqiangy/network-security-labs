1. Lab Purpose

The purpose of this lab is to help students understand how digital certificates are issued and verified. Students will create a private root CA, use it to sign a server certificate for Ubuntu-Target, and verify the certificate from Kali-Lab using the private CA certificate as the trust anchor.

Main goal: Students should understand that a certificate is trusted because a trusted CA signs it, and that private CA trust is local to the systems where the CA certificate is explicitly trusted.

2. Learning Objectives

  1. Explain the role of a Certificate Authority in public key infrastructure.
  2. Create a private root CA key and self-signed root certificate with OpenSSL.
  3. Generate a server private key and certificate signing request.
  4. Add Subject Alternative Name values for a lab server certificate.
  5. Sign a server certificate using the private CA.
  6. Inspect certificate issuer, subject, validity, key usage, and SAN fields.
  7. Verify a certificate chain using the CA certificate.
  8. Run a local TLS test service using the signed certificate.
  9. Use OpenSSL client verification to confirm trust and identify failures.
  10. Explain OS-specific differences for certificate stores and OpenSSL commands.

3. Lab Scenario

A small organization wants to test internal TLS certificates for lab servers without purchasing public certificates. You are asked to create a private CA, issue a certificate to an internal Ubuntu server, and prove that the certificate can be verified by a system that trusts the private CA certificate.

6. Safety Rules

  1. Create and use the private CA only inside the isolated lab environment.
  2. Do not install this lab CA certificate into production trust stores.
  3. Do not use lab CA keys or lab certificates on real systems.
  4. Do not submit private keys or passphrases.
  5. Do not copy or reuse real organization certificates or keys.
  6. Do not expose the temporary TLS server outside the host-only/private lab network.
  7. Stop the temporary TLS test server before finishing the lab.
Important: A private CA can issue certificates that systems may trust if the CA certificate is installed. Treat CA private keys as highly sensitive, even in a lab.

7. Key Concepts

Certificate Authority

A Certificate Authority, or CA, signs certificates. Systems trust a certificate when it chains back to a CA certificate that the system trusts.

Root CA Certificate

A root CA certificate is self-signed and acts as a trust anchor. In this lab, the root CA is private and trusted only when explicitly referenced with -CAfile.

Certificate Signing Request

A CSR contains a public key and identity information for the requested certificate. A CA signs the CSR to issue a certificate.

Subject Alternative Name

Modern TLS verification relies on Subject Alternative Name, or SAN, values. The certificate should include the DNS name or IP address clients use to connect.

Public vs. Private Certificate Use

Certificate TypeTypical UseTrust Model
Public CA certificatePublic websites and internet servicesBrowsers and OSs already trust many public roots
Private CA certificateInternal labs, organizations, development, private servicesClients must be configured to trust the private CA
Self-signed server certificateTesting onlyUsually not trusted unless manually accepted

Part 5 — Create a Self-Signed Root CA Certificate

On Kali-Lab:

openssl req -x509 -new -nodes \
  -key ~/Lab4-3-PrivateCA/ca/private_lab_root_ca.key.pem \
  -sha256 -days 3650 \
  -out ~/Lab4-3-PrivateCA/ca/private_lab_root_ca.cert.pem \
  -subj "/C=US/ST=Alabama/L=Madison/O=Network Security Lab/OU=Private Lab CA/CN=Network Security Lab Root CA"

openssl x509 -in ~/Lab4-3-PrivateCA/ca/private_lab_root_ca.cert.pem -noout -subject -issuer -dates -fingerprint -sha256
sha256sum ~/Lab4-3-PrivateCA/ca/private_lab_root_ca.cert.pem > ~/Lab4-3-PrivateCA/ca/private_lab_root_ca.cert.pem.sha256

Required Screenshot 5: Self-signed private root CA certificate created and inspected.

Part 7 — Create a Server CSR Configuration with SAN

On Kali-Lab, replace UBUNTU-IP-ADDRESS with the actual Ubuntu-Target IP address:

cat > ~/Lab4-3-PrivateCA/server-cert/ubuntu_server_csr.cnf <<'EOF'
[ req ]
default_bits       = 3072
prompt             = no
default_md         = sha256
distinguished_name = dn
req_extensions     = req_ext

[ dn ]
C  = US
ST = Alabama
L  = Madison
O  = Network Security Lab
OU = Internal Lab Server
CN = ubuntu-target.local

[ req_ext ]
subjectAltName = @alt_names

[ alt_names ]
DNS.1 = ubuntu-target.local
DNS.2 = ubuntu-target
IP.1  = UBUNTU-IP-ADDRESS
EOF

sed -i 's/UBUNTU-IP-ADDRESS/YOUR-ACTUAL-UBUNTU-IP/' ~/Lab4-3-PrivateCA/server-cert/ubuntu_server_csr.cnf
cat ~/Lab4-3-PrivateCA/server-cert/ubuntu_server_csr.cnf
Important: Edit the file so IP.1 contains the actual Ubuntu-Target IP. If the SAN IP does not match, client verification by IP may fail.

Required Screenshot 7: CSR configuration file with correct SAN values.

Part 9 — Create Server Certificate Extension File

On Kali-Lab, again replace the IP value with the actual Ubuntu-Target IP:

cat > ~/Lab4-3-PrivateCA/server-cert/ubuntu_server_cert.ext <<'EOF'
authorityKeyIdentifier=keyid,issuer
basicConstraints=CA:FALSE
keyUsage=digitalSignature,keyEncipherment
extendedKeyUsage=serverAuth
subjectAltName=@alt_names

[alt_names]
DNS.1=ubuntu-target.local
DNS.2=ubuntu-target
IP.1=UBUNTU-IP-ADDRESS
EOF

sed -i 's/UBUNTU-IP-ADDRESS/YOUR-ACTUAL-UBUNTU-IP/' ~/Lab4-3-PrivateCA/server-cert/ubuntu_server_cert.ext
cat ~/Lab4-3-PrivateCA/server-cert/ubuntu_server_cert.ext

Required Screenshot 9: Server certificate extension file created.

Part 10 — Sign the Server Certificate with the Private CA

On Kali-Lab:

openssl x509 -req \
  -in ~/Lab4-3-PrivateCA/server-cert/ubuntu_target_server.csr.pem \
  -CA ~/Lab4-3-PrivateCA/ca/private_lab_root_ca.cert.pem \
  -CAkey ~/Lab4-3-PrivateCA/ca/private_lab_root_ca.key.pem \
  -CAcreateserial \
  -out ~/Lab4-3-PrivateCA/server-cert/ubuntu_target_server.cert.pem \
  -days 365 -sha256 \
  -extfile ~/Lab4-3-PrivateCA/server-cert/ubuntu_server_cert.ext

openssl x509 -in ~/Lab4-3-PrivateCA/server-cert/ubuntu_target_server.cert.pem -noout -subject -issuer -dates
openssl x509 -in ~/Lab4-3-PrivateCA/server-cert/ubuntu_target_server.cert.pem -noout -text | grep -A3 "Subject Alternative Name"

Required Screenshot 10: Server certificate signed by the private CA.

11. OS-Specific Differences

Kali-Lab

Kali is used as the private CA workstation, certificate signing system, and TLS verification client.

openssl genpkey -algorithm RSA -pkeyopt rsa_keygen_bits:4096 -out ca.key.pem
openssl req -x509 -new -key ca.key.pem -out ca.cert.pem
openssl x509 -req -in server.csr.pem -CA ca.cert.pem -CAkey ca.key.pem -out server.cert.pem
openssl verify -CAfile ca.cert.pem server.cert.pem
openssl s_client -connect IP:8443 -CAfile ca.cert.pem
