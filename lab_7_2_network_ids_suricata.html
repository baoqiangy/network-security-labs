1. Lab Purpose

The purpose of this lab is to introduce network intrusion detection using Suricata. Students will deploy Suricata on Ubuntu-Target, configure the monitored network interface, generate safe local network traffic from Kali-Lab, review IDS logs, and create a local custom rule that detects a harmless lab-specific HTTP request.

Main goal: Students should understand how an IDS observes traffic, generates alerts, records metadata, and supports security monitoring without directly blocking traffic.

2. Learning Objectives

  1. Define the role of a network IDS in a security monitoring architecture.
  2. Explain the difference between IDS and IPS behavior.
  3. Install and verify Suricata on Ubuntu-Target.
  4. Identify the correct network interface for Suricata monitoring.
  5. Review and validate Suricata configuration using suricata -T.
  6. Start Suricata and confirm log files are generated.
  7. Generate safe local traffic using ping, curl, and selected-port scans.
  8. Review Suricata fast.log, eve.json, and stats.log.
  9. Create and test a local custom Suricata rule.
  10. Document IDS findings, false positives, limitations, and tuning recommendations.

3. Lab Scenario

Your organization wants to improve visibility into network activity. You are asked to deploy a lightweight IDS sensor on an Ubuntu system in a local lab network. The sensor should observe safe traffic between Kali-Lab and Ubuntu-Target, record alerts and flow events, and demonstrate how a custom detection rule can identify a known lab-specific HTTP request.

6. Safety Rules

  1. Generate traffic only against your assigned local Ubuntu-Target VM.
  2. Do not scan public IP addresses, school systems, classmates' systems, or production networks.
  3. Use only safe, low-volume test traffic.
  4. Do not run exploit tools or denial-of-service tests.
  5. Do not download suspicious malware samples or unsafe rule packs.
  6. Do not submit sensitive tokens, private keys, or passwords from logs.
  7. Keep the lab network host-only/private or instructor-controlled.
  8. Document exactly what traffic you generated.
Important: This is a defensive monitoring lab. The goal is to observe and interpret local lab traffic, not to attack systems.

7. Key Concepts

Network IDS

A network intrusion detection system observes network traffic and generates alerts when traffic matches known signatures, suspicious behavior, or configured rules.

IDS vs. IPS

An IDS detects and alerts. An IPS can actively block or modify traffic. This lab uses Suricata in IDS-style monitoring mode.

Rules and Signatures

Suricata rules describe traffic patterns to detect. A rule can match protocol, source, destination, ports, message content, and metadata.

EVE JSON

eve.json is Suricata's structured JSON event log. It can contain alerts, flow records, DNS events, HTTP events, TLS metadata, and statistics depending on configuration.

False Positives and Tuning

An IDS can generate alerts that are not truly malicious. Security analysts must tune rules and interpret context to reduce noise.

Part 15 — Create a Local Custom Rule

Create a harmless custom rule that detects a lab-specific HTTP request containing lab72-suricata-test. On Ubuntu-Target:

sudo mkdir -p /etc/suricata/rules
echo 'alert http any any -> any any (msg:"LAB 7.2 Custom HTTP Test String Detected"; content:"lab72-suricata-test"; http_uri; sid:10000072; rev:1; classtype:policy-violation;)' | sudo tee /etc/suricata/rules/local.rules

sudo cat /etc/suricata/rules/local.rules | tee ~/Lab7-2-Suricata/rules/local-rules-created.txt

Make sure local.rules is included in suricata.yaml. Check:

sudo grep -n "local.rules" /etc/suricata/suricata.yaml || true

If local.rules is not included, add it under the rule-files list in /etc/suricata/suricata.yaml:

sudo nano /etc/suricata/suricata.yaml

Example rule-files section should include:

rule-files:
  - suricata.rules
  - local.rules

15. Troubleshooting Guide

ProblemLikely CauseFix
Suricata does not installNo repository access or package issueUse instructor-provided package/VM or document repository error.
suricata -T failsYAML syntax error or bad ruleReview the error line, restore backup if needed, and retest.
Suricata starts but no logs appearWrong interface or no trafficVerify interface name and generate traffic from Kali.
No custom alert appearslocal.rules not included, rule syntax issue, no HTTP traffic, or wrong interfaceCheck config, run suricata -T, restart Suricata, and verify traffic reaches sensor.
jq command failsjq not installed or eve.json emptyInstall jq if allowed or use tail/grep on eve.json.
curl to Ubuntu failsNo web server running or firewall blockingInstall/start Apache if allowed or document failed HTTP service.
Nmap shows filtered ports but no IDS alertRules do not detect that pattern by defaultDocument that not all scans generate alerts without appropriate rules.
Permission denied reading logsSuricata logs require elevated permissionsUse sudo to read logs.
Interface name differs from instructionsDifferent OS/VM network namingUse ip -brief addr to identify the correct interface.
