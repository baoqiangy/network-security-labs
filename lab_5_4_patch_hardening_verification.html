1. Lab Purpose

The purpose of this lab is to teach students how to verify whether basic patching and hardening controls are present on a local Linux server. Students will collect a baseline, review package update status, check exposed services, confirm firewall rules, evaluate SSH configuration, inspect optional web/TLS settings, perform a post-hardening scan, and document remaining risk.

Main goal: Students should learn that hardening is not just applying changes. It also requires verification evidence showing that the intended controls are actually working.

2. Learning Objectives

  1. Explain the difference between patching, hardening, and verification.
  2. Collect a system baseline from Ubuntu-Target.
  3. Review package update status and identify pending updates.
  4. Safely apply updates if permitted by the instructor.
  5. Identify exposed services using local commands and remote scans.
  6. Verify firewall rules and confirm that only intended ports are reachable.
  7. Review SSH hardening settings and identify risky defaults.
  8. Review web and TLS hardening controls if Apache/HTTPS is present.
  9. Compare before-and-after scan evidence.
  10. Create a patch and hardening verification report with remediation priorities.

3. Lab Scenario

You are a security analyst assigned to review a local Ubuntu server before it is approved for continued use in a small organization. Previous labs configured services such as SSH, firewall rules, Apache, and HTTPS. Your job is to verify whether the system is patched, whether unnecessary services are exposed, and whether basic hardening controls are in place.

6. Safety Rules

  1. Verify and harden only the assigned Ubuntu-Target VM.
  2. Do not scan or change public systems, school systems, classmates' systems, or production systems.
  3. Take screenshots and save evidence before and after changes.
  4. Do not remove services unless the instructor explicitly allows it.
  5. Do not lock yourself out of SSH; verify access before closing existing sessions.
  6. Do not apply major upgrades or distribution upgrades unless instructed.
  7. Keep the virtual network host-only/private.
  8. Document every hardening change made.
Important: Hardening can accidentally break access or services. In this lab, changes are limited to the local VM and should be verified immediately.

7. Key Concepts

Patching

Patching means applying software updates that fix bugs, security vulnerabilities, and stability issues. Patch verification checks whether updates are pending or successfully installed.

Hardening

Hardening means reducing attack surface and strengthening configuration. Examples include disabling unnecessary services, restricting firewall rules, and enforcing safer SSH settings.

Verification

Verification means collecting evidence that a control is working. For example, a firewall rule is not verified just because it exists; it should also be tested from another system.

Attack Surface

Attack surface refers to the services, ports, accounts, software, and interfaces that could be targeted. A smaller attack surface is easier to protect and monitor.

Part 12 — Verify SSH Hardening Settings

On Ubuntu-Target:

sudo sshd -T | grep -Ei 'permitrootlogin|passwordauthentication|maxauthtries|pubkeyauthentication|x11forwarding|allowusers|clientaliveinterval|clientalivecountmax'   | tee ~/Lab5-4-HardeningVerify/ssh-hardening/sshd-effective-settings.txt

Part 13 — Optional SSH Hardening Change if Permitted

Avoid lockout: Keep your current SSH session open. Test new SSH access before closing existing connections.

If instructed to make SSH hardening changes, back up the configuration first:

sudo cp /etc/ssh/sshd_config /etc/ssh/sshd_config.lab5-4.bak
sudo ls -l /etc/ssh/sshd_config.lab5-4.bak

Example safe additions for a lab server:

sudo tee /etc/ssh/sshd_config.d/99-lab-hardening.conf > /dev/null <<'EOF'
PermitRootLogin no
MaxAuthTries 3
X11Forwarding no
ClientAliveInterval 300
ClientAliveCountMax 2
EOF

sudo sshd -t
sudo systemctl reload ssh
sudo sshd -T | grep -Ei 'permitrootlogin|maxauthtries|x11forwarding|clientaliveinterval|clientalivecountmax'   | tee ~/Lab5-4-HardeningVerify/ssh-hardening/sshd-effective-settings-after-change.txt
