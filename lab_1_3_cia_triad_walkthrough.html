1. Lab Purpose

The purpose of this lab is to give students a practical walkthrough of the CIA triad: confidentiality, integrity, and availability. Students will use simple Linux commands to protect a file from unauthorized viewing, detect file changes using hashes, and test whether a basic service is available over the lab network.

Main goal: Students should connect the CIA triad to real technical actions: access control protects confidentiality, hashing supports integrity, and service testing supports availability.

2. Learning Objectives

  1. Define confidentiality, integrity, and availability in practical terms.
  2. Use Linux file permissions to demonstrate confidentiality.
  3. Create test users and compare authorized and unauthorized file access.
  4. Use SHA-256 hashing to create an integrity baseline.
  5. Modify a file and detect the integrity change using hash comparison.
  6. Start a simple web service and verify availability from another VM.
  7. Stop the service and observe the effect on availability.
  8. Document evidence for confidentiality, integrity, and availability.
  9. Explain how CIA triad concepts apply to network security controls.
  10. Explain differences across Windows, Intel Mac, Apple Silicon Mac, Kali, and Ubuntu environments.

3. Lab Scenario

A small organization has asked you to demonstrate the three core security goals of the CIA triad using a safe lab environment. You will protect a sensitive file from unauthorized users, verify whether a file has changed, and test whether a basic internal service is reachable from a security workstation.

6. Safety Rules

  1. Use only the files and users created for this lab.
  2. Do not modify operating system files or real user files.
  3. Do not collect or submit passwords or password hashes.
  4. Do not run denial-of-service tools or availability attacks.
  5. Availability testing in this lab means starting, stopping, and checking a temporary local service only.
  6. Keep the lab VMs on host-only/private networking unless the instructor authorizes temporary Internet access.
  7. Stop the temporary web service at the end of the lab.
Important: This lab demonstrates concepts safely. It does not involve attacking systems or disrupting real services.

7. Key Concepts

Confidentiality

Confidentiality means information is accessible only to authorized users or systems. In this lab, confidentiality is demonstrated with Linux file ownership, groups, and permissions.

Integrity

Integrity means information remains accurate and unchanged unless changed by authorized activity. In this lab, integrity is demonstrated with SHA-256 hashes and file comparison.

Availability

Availability means systems and data are accessible when needed. In this lab, availability is demonstrated by starting, testing, stopping, and retesting a temporary web service.

Part 4 — Confidentiality Demonstration: Create Users and Group

On Ubuntu-Target, create two lab users and one security group:

sudo groupadd cia_confidential
sudo useradd -m -s /bin/bash authorizeduser
sudo useradd -m -s /bin/bash unauthorizeduser
sudo passwd authorizeduser
sudo passwd unauthorizeduser
sudo usermod -aG cia_confidential authorizeduser

Verify users and group membership:

id authorizeduser
id unauthorizeduser
getent group cia_confidential

Save the evidence:

id authorizeduser > ~/Lab1-3-CIATriad/confidentiality/authorizeduser-id.txt
id unauthorizeduser > ~/Lab1-3-CIATriad/confidentiality/unauthorizeduser-id.txt
getent group cia_confidential > ~/Lab1-3-CIATriad/confidentiality/cia-confidential-group.txt

Required Screenshot 4: Users and group created.

Part 5 — Confidentiality Demonstration: Create and Protect a Sensitive File

On Ubuntu-Target:

sudo mkdir -p /opt/cia-lab
sudo bash -c 'cat > /opt/cia-lab/sensitive_report.txt << EOF
Sensitive Report
Owner: Security Team
Classification: Confidential
Message: This file should be readable only by authorized users.
EOF'

sudo chown root:cia_confidential /opt/cia-lab/sensitive_report.txt
sudo chmod 640 /opt/cia-lab/sensitive_report.txt
ls -l /opt/cia-lab/sensitive_report.txt

Save evidence:

ls -l /opt/cia-lab/sensitive_report.txt > ~/Lab1-3-CIATriad/confidentiality/sensitive-file-permissions.txt

Required Screenshot 5: Sensitive file ownership and permissions.

Part 6 — Confidentiality Demonstration: Test Authorized and Unauthorized Access

On Ubuntu-Target, test access as the authorized user:

sudo -u authorizeduser cat /opt/cia-lab/sensitive_report.txt

Expected result: the file content is displayed.

Test access as the unauthorized user:

sudo -u unauthorizeduser cat /opt/cia-lab/sensitive_report.txt

Expected result: Permission denied.

Save both results:

sudo -u authorizeduser cat /opt/cia-lab/sensitive_report.txt > ~/Lab1-3-CIATriad/confidentiality/authorized-access-result.txt

sudo -u unauthorizeduser cat /opt/cia-lab/sensitive_report.txt > ~/Lab1-3-CIATriad/confidentiality/unauthorized-access-result.txt 2>&1 || true

Required Screenshot 6: Authorized user succeeds and unauthorized user fails.

Part 9 — Integrity Demonstration: Modify the File and Detect the Change

On Ubuntu-Target:

cd ~/Lab1-3-CIATriad/integrity/files
echo "Status: Modified without approval" >> inventory.txt

15. Troubleshooting Guide

ProblemLikely CauseFix
Kali cannot ping UbuntuWrong IP or VMs are on different networksVerify ip -brief addr and host-only/private network settings.
groupadd says group already existsThe lab was attempted beforeContinue using the existing group, or ask the instructor before deleting anything.
useradd says user already existsThe lab user was already createdContinue using the existing user, or choose an instructor-approved alternate username.
Authorized user cannot read the fileUser is not in the group or permissions are wrongCheck id authorizeduser, getent group cia_confidential, and ls -l.
Unauthorized user can read the fileFile permissions are too openUse sudo chmod 640 /opt/cia-lab/sensitive_report.txt.
Hash verification says file not foundThe hash file is in a different folder from the file being checkedRun sha256sum -c from the folder containing both files.
Kali cannot reach the web serviceWrong IP, server not running, or port not listeningCheck python3 -m http.server terminal and ss on Ubuntu.
Port 8080 remains open after cleanupPython server is still runningPress Ctrl + C in the server terminal and retest.
