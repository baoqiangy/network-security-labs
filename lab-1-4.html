Lab 1.4 — Network Troubleshooting and Evidence Collection

Network Security hands-on lab based on the Lab 1.1 virtual environment. Students troubleshoot common connectivity problems, collect command-line evidence, compare symptoms with likely causes, and write a professional troubleshooting report.

Course: Network Security Duration: 1 class period VMs: Kali-Lab + Ubuntu-Target Tools: ip, ping, traceroute, ss, curl, SSH, journalctl

1. Lab Purpose

This lab teaches students how to troubleshoot network problems systematically. Students will verify the known-good baseline, intentionally observe or simulate common failure conditions, identify likely causes, restore normal connectivity, and document evidence.

Kali-Lab ───── Host-only / Private Network ───── Ubuntu-Target
Main goal: Students should be able to move from symptom to evidence to likely cause to corrective action, rather than guessing.

2. Learning Objectives

  1. Apply a structured troubleshooting process to a small virtual network.
  2. Verify baseline connectivity between Kali-Lab and Ubuntu-Target.
  3. Use ip addr, ip route, ping, ss, curl, and ssh to collect evidence.
  4. Distinguish between host reachability problems and service-level problems.
  5. Identify symptoms caused by wrong IP address, wrong network adapter, stopped service, firewall block, or incorrect port.
  6. Record before-and-after troubleshooting evidence.
  7. Restore the lab environment to a working state.
  8. Create a concise network troubleshooting report.
  9. Explain OS-specific differences for Windows, Intel Mac, Apple Silicon Mac, Kali, and Ubuntu environments.

3. Lab Scenario

A student reports that Kali-Lab cannot reliably connect to Ubuntu-Target. Sometimes ping works, but SSH or web testing fails. Your task is to diagnose the problem using evidence, determine the likely cause, and restore the lab network to a known-good state.

6. Safety Rules

  1. Use only Kali-Lab and Ubuntu-Target.
  2. Keep both VMs on the isolated host-only/private virtual network.
  3. Do not use bridged networking.
  4. Do not scan or test public systems, school networks, home routers, cloud systems, or classmates’ machines.
  5. Do not disable the host computer network or host firewall.
  6. Do not submit screenshots containing passwords or sensitive personal information.
  7. Restore the lab to a known-good working state before submitting.

7. Key Concepts

Structured Troubleshooting

Good troubleshooting follows a process: observe the symptom, collect evidence, identify the likely cause, apply a fix, validate the result, and document what changed.

Host Reachability vs. Service Reachability

TestWhat It ProvesWhat It Does Not Prove
pingThe target host can respond to ICMPSSH or web services are working
ssh user@IPSSH service is reachable and login can workOther services are working
curl http://IP:8080The web service is reachable on port 8080SSH is working
ss -tulpenWhich services are listening locallyWhether a remote host can reach them through a firewall

Evidence-Based Notes

Every troubleshooting conclusion should be supported by command output, screenshots, saved log files, or a clear before-and-after comparison.

mkdir -p ~/Lab1-4-Troubleshooting/{baseline,symptoms,tests,fixes,logs,report}

Part 2 — Record Baseline System Information

On Kali-Lab:

hostname
uname -m
ip -brief addr
ip route

Save:

hostname > ~/Lab1-4-Troubleshooting/baseline/kali-hostname.txt

ip -brief addr > ~/Lab1-4-Troubleshooting/baseline/kali-ip-brief.txt
ip route > ~/Lab1-4-Troubleshooting/baseline/kali-ip-route.txt

On Ubuntu-Target:

hostname
uname -m

Save:

hostname > ~/Lab1-4-Troubleshooting/baseline/ubuntu-hostname.txt

ip -brief addr > ~/Lab1-4-Troubleshooting/baseline/ubuntu-ip-brief.txt
ip route > ~/Lab1-4-Troubleshooting/baseline/ubuntu-ip-route.txt

Required Screenshot 2: Baseline IP, route, hostname, and architecture information.

Part 3 — Verify Baseline Ping Connectivity

From Kali-Lab:

ping -c 4 UBUNTU-IP-ADDRESS | tee ~/Lab1-4-Troubleshooting/tests/kali-to-ubuntu-ping-baseline.txt

From Ubuntu-Target:

ping -c 4 KALI-IP-ADDRESS | tee ~/Lab1-4-Troubleshooting/baseline/ubuntu-to-kali-ping-baseline.txt

Required Screenshot 3: Successful ping in both directions.

Part 4 — Verify SSH Service and SSH Connectivity

On Ubuntu-Target, check SSH:

sudo systemctl status ssh --no-pager

If SSH is not running:

sudo systemctl enable ssh
sudo systemctl start ssh

Save SSH status:

sudo systemctl status ssh --no-pager > ~/Lab1-4-Troubleshooting/service-info/ssh-status-baseline.txt

From Kali-Lab, test SSH:

ssh YOUR-UBUNTU-USERNAME@UBUNTU-IP-ADDRESS
whoami
hostname
exit

Save a note on Kali:

echo "Baseline SSH test succeeded." > ~/Lab1-4-Troubleshooting/tests/ssh-baseline-result.txt

Required Screenshot 4: SSH service status and successful SSH login.

Part 5 — Verify Listening Services

On Ubuntu-Target:

sudo ss -tulpen

Save:

sudo ss -tulpen > ~/Lab1-4-Troubleshooting/service-info/listening-services-baseline.txt

Students should identify whether port 22 is listening.

Required Screenshot 5: Ubuntu listening services showing SSH.

Part 6 — Troubleshooting Case 1: Wrong IP Address

From Kali-Lab, intentionally ping an incorrect or unused IP address on the same virtual network. Example:

ping -c 4 WRONG-OR-UNUSED-IP-ADDRESS | tee ~/Lab1-4-Troubleshooting/symptoms/wrong-ip-ping-failure.txt

Then ping the correct Ubuntu IP address:

ping -c 4 UBUNTU-IP-ADDRESS | tee ~/Lab1-4-Troubleshooting/fixes/correct-ip-ping-success.txt

Document the lesson:

echo "A wrong IP address can look like a network failure. The correct IP restored connectivity." \
> ~/Lab1-4-Troubleshooting/fixes/wrong-ip-lesson.txt

Required Screenshot 6: Wrong-IP failure and correct-IP success.

Part 7 — Troubleshooting Case 2: Host Reachable but Service Not Running

On Ubuntu-Target, make sure no temporary web server is running on port 8080:

sudo ss -tulpen | grep 8080

From Kali-Lab, test port 8080 before starting the web service:

curl --connect-timeout 5 -v http://UBUNTU-IP-ADDRESS:8080 \
> ~/Lab1-4-Troubleshooting/symptoms/http-8080-service-down.txt 2>&1

Now start the web service on Ubuntu-Target:

mkdir -p ~/Lab1-4-WebTest
cd ~/Lab1-4-WebTest
echo "Lab 1.4 troubleshooting web test" > index.html
python3 -m http.server 8080 --bind 0.0.0.0

Leave the server running. From Kali-Lab, test again:

curl http://UBUNTU-IP-ADDRESS:8080 | tee ~/Lab1-4-Troubleshooting/fixes/http-8080-service-up.txt

Required Screenshot 7: Port 8080 failure before service start and success after service start.

Part 8 — Troubleshooting Case 3: Host Reachable but SSH Stopped

Important: Only perform this step from the Ubuntu console. Do not stop SSH if you do not have direct access to the Ubuntu VM window.

On Ubuntu-Target, stop SSH temporarily:

sudo systemctl stop ssh
sudo systemctl status ssh --no-pager

From Kali-Lab, confirm ping still works:

ping -c 4 UBUNTU-IP-ADDRESS

Then test SSH:

ssh YOUR-UBUNTU-USERNAME@UBUNTU-IP-ADDRESS

Save the failed SSH symptom on Kali:

echo "SSH failed while ping still worked because SSH service was stopped." \
> ~/Lab1-4-Troubleshooting/symptoms/ssh-stopped-symptom.txt

Restore SSH on Ubuntu-Target:

sudo systemctl start ssh
sudo systemctl status ssh --no-pager

Test SSH again from Kali-Lab:

ssh YOUR-UBUNTU-USERNAME@UBUNTU-IP-ADDRESS
hostname
exit

Required Screenshot 8: SSH stopped symptom and restored SSH success.

Part 9 — Troubleshooting Case 4: Wrong Port

From Kali-Lab, try SSH on the wrong port:

ssh -p 2222 YOUR-UBUNTU-USERNAME@UBUNTU-IP-ADDRESS

This should fail unless a service is intentionally configured on port 2222.

Now test the correct SSH port:

ssh -p 22 YOUR-UBUNTU-USERNAME@UBUNTU-IP-ADDRESS
exit

Save a note:

echo "Wrong port testing showed that host reachability does not mean every port is open." \
> ~/Lab1-4-Troubleshooting/fixes/wrong-port-lesson.txt

Required Screenshot 9: Wrong port failure and correct port success.

Part 10 — Troubleshooting Case 5: Verify Local Service State with ss

On Ubuntu-Target, check listening services again:

sudo ss -tulpen

Save:

sudo ss -tulpen > ~/Lab1-4-Troubleshooting/service-info/listening-services-after-tests.txt

Students should compare this file with the baseline listening-service file.

Required Screenshot 10: Listening services after troubleshooting tests.

Part 11 — Review SSH Logs on Ubuntu

On Ubuntu-Target, review recent SSH logs:

Part 13 — Restore Final Known-Good State

On Ubuntu-Target, make sure SSH is running:

sudo ss -tulpen | grep 8080

From Kali-Lab, verify final ping and SSH:

ping -c 4 UBUNTU-IP-ADDRESS
ssh YOUR-UBUNTU-USERNAME@UBUNTU-IP-ADDRESS

exit

SSH connection refusedSSH service is stoppedRun sudo systemctl start ssh on Ubuntu.
