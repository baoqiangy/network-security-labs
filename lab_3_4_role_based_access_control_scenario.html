1. Lab Purpose

The purpose of this lab is to give students practical experience with role-based access control. Students will translate job responsibilities into Linux groups, directory permissions, file permissions, and access tests. The lab emphasizes least privilege, separation of duties, and verification of both allowed and denied actions.

Main goal: Students should learn that RBAC starts with job roles and business needs, then becomes technical enforcement through users, groups, permissions, and controlled privilege assignment.

2. Learning Objectives

  1. Define role-based access control and explain why organizations use it.
  2. Translate job responsibilities into access-control roles.
  3. Create Linux groups that represent organizational roles.
  4. Create users and assign them to appropriate role groups.
  5. Create shared folders for administrative, help desk, analyst/student, guest, and auditor access.
  6. Configure read/write, read-only, and denied access using ownership, groups, permissions, and ACLs where needed.
  7. Test allowed and denied access as different users.
  8. Design a limited administrative privilege example without granting broad root access.
  9. Document an RBAC matrix and access-test results.
  10. Explain OS-specific differences across Windows, Intel Mac, Apple Silicon Mac, Kali, and Ubuntu environments.

3. Lab Scenario

A small organization has five types of users: administrators, help desk staff, analysts/students, guests, and auditors. Each role needs different access. Administrators need broad management access, help desk users need limited support access, analysts need write access to project files, guests should have very limited read-only access, and auditors should be able to review selected evidence without modifying it. You will design and test a role-based access-control model for this organization.

6. Safety Rules

  1. Perform RBAC changes only on your assigned Ubuntu-Target VM.
  2. Do not modify real user accounts on your host computer or production systems.
  3. Do not delete system users, system groups, or your own sudo access.
  4. Do not submit passwords or password hashes.
  5. Use only temporary lab users and folders created in this lab.
  6. Use visudo or a sudoers drop-in file for privilege tests; do not edit /etc/sudoers directly with a normal text editor.
  7. Ask the instructor before cleanup if later labs may reuse the users or groups.
  8. Keep the lab network host-only/private.
Important: RBAC mistakes can accidentally grant too much access. Every allow rule should be tested, and every expected denial should also be tested.

7. Key Concepts

Role-Based Access Control

Role-Based Access Control, or RBAC, assigns permissions to roles rather than directly to individual users. Users receive permissions by becoming members of role groups.

Least Privilege

Least privilege means users should receive only the access they need to perform their assigned tasks and no more.

Separation of Duties

Separation of duties prevents one role from having excessive control over sensitive processes. For example, an auditor may review logs and evidence but should not modify them.

Part 15 — Create a Limited Help Desk Sudo Example

Use a limited test command only. This demonstrates controlled privilege assignment without granting broad administrator rights.

On Ubuntu-Target, allow help desk users to run only /usr/bin/id as a safe sudo demonstration:

echo "%rbac_helpdesk ALL=(root) NOPASSWD: /usr/bin/id" | sudo tee /etc/sudoers.d/lab34-helpdesk-id
sudo chmod 440 /etc/sudoers.d/lab34-helpdesk-id
sudo visudo -cf /etc/sudoers.d/lab34-helpdesk-id

Test allowed and denied commands:

sudo -u helpdesk_user sudo /usr/bin/id
sudo -u helpdesk_user sudo /usr/bin/apt update

Expected result: helpdesk_user can run id with sudo but should not be allowed to run broad commands such as apt update.

Save evidence:

sudo -u helpdesk_user sudo /usr/bin/id > ~/Lab3-4-RBAC/sudo-tests/helpdesk-limited-sudo-id-allowed.txt 2>&1 || true
sudo -u helpdesk_user sudo /usr/bin/apt update > ~/Lab3-4-RBAC/sudo-tests/helpdesk-apt-denied.txt 2>&1 || true
sudo cat /etc/sudoers.d/lab34-helpdesk-id > ~/Lab3-4-RBAC/sudo-tests/helpdesk-limited-sudo-rule.txt

15. Troubleshooting Guide

ProblemLikely CauseFix
User or group already existsThe lab was attempted beforeDocument it and continue, or ask the instructor before deleting anything.
User cannot access assigned resourceWrong group membership, ownership, or permissionsCheck id username, getent group groupname, and ls -ld resource.
User can access a resource that should be deniedPermissions are too open or ACL grants accessCheck ls -l, ls -ld, and getfacl.
setfacl not foundACL package may not be installedAsk the instructor before installing acl, or document that ACL testing was unavailable.
Auditor cannot read project fileMissing directory execute permission or missing file read ACLUse ACLs on both directory and file: rx for directory and r-- for file.
Auditor can write when read-only was expectedACL or group permissions are too broadReview getfacl and remove write permission.
Sudoers test fails with syntax errorIncorrect sudoers syntaxUse sudo visudo -cf /etc/sudoers.d/lab34-helpdesk-id and fix the file.
scp failsWrong username, IP, path, or SSH issueVerify SSH access and the evidence folder path.
Cleanup removes needed usersCleanup performed before instructor approvalAsk before cleanup if later labs will reuse these accounts.
