Network Security hands-on lab based on the Lab 6.1 pfSense firewall environment. 1. Lab Purpose

The purpose of this lab is to help students design and test a demilitarized zone, or DMZ, using pfSense. Students will add a third firewall interface, separate an internal LAN from a semi-public DMZ network, host a basic web service in the DMZ, and verify that firewall rules allow only the intended traffic.

Main goal: Students should understand that a DMZ is used to isolate externally reachable or higher-risk services from the trusted internal LAN.

2. Learning Objectives

  1. Explain the purpose of a DMZ in network security architecture.
  2. Extend a pfSense firewall from two interfaces to three interfaces.
  3. Create separate WAN, LAN, and DMZ virtual networks in VMware.
  4. Assign and configure a pfSense DMZ interface.
  5. Place a web server on the DMZ network.
  6. Create firewall rules that allow LAN-to-DMZ management/testing traffic.
  7. Create firewall rules that prevent DMZ-initiated access to the LAN.
  8. Allow limited DMZ-to-WAN traffic for updates or DNS if permitted.
  9. Test allowed and blocked traffic using ping, curl, and nmap.
  10. Review pfSense firewall logs and document rule effectiveness.

3. Lab Scenario

A small organization wants to publish an internal test web server without placing it directly on the trusted LAN. You are asked to design a DMZ network using pfSense. The web server should be reachable from the trusted LAN for testing, but the DMZ server should not be allowed to initiate access to the LAN. If upstream access is available, the DMZ server may be allowed limited outbound access for updates and DNS.

6. Safety Rules

  1. Use only isolated virtual networks for the LAN and DMZ.
  2. Do not bridge the DMZ or LAN to a real public, school, or home network.
  3. Do not expose pfSense WebGUI to the WAN or DMZ interface.
  4. Do not create broad WAN-to-LAN or WAN-to-DMZ rules unless explicitly instructed.
  5. Do not scan systems outside the local lab.
  6. Do not run exploit or denial-of-service tests.
  7. Take a pfSense snapshot before adding the third interface or changing firewall rules.
  8. Record each rule you add and test it immediately.
Important: A DMZ is meant to reduce risk. Overly broad rules can defeat the purpose of segmentation.

7. Key Concepts

DMZ

A DMZ is a separate network segment used for services that need more exposure than internal LAN systems. A DMZ limits the damage if a public-facing or semi-public service is compromised.

Segmentation

Segmentation divides a network into zones with different trust levels. Firewall rules control traffic between those zones.

Interface-Based Firewall Rules

pfSense rules are evaluated on the interface where traffic enters the firewall. For example, rules controlling DMZ-initiated traffic are created on the DMZ interface.

Least Privilege Network Access

Firewall rules should allow only what is needed. In a DMZ, this usually means allowing specific service ports while blocking access to internal LAN resources.

Part 2 — Create or Identify the DMZ Virtual Network

  1. Open Virtual Network Editor as administrator.
  2. Keep the WAN as VMnet8 NAT.
  3. Keep the LAN as the Lab 6.1 host-only network, such as VMnet2.

  1. Use the compatible firewall option assigned by the instructor.
  2. Create or identify separate virtual networks for LAN and DMZ if the platform supports it.

Part 6 — Move Ubuntu-Target to the DMZ Network

  1. Power off Ubuntu-Target.
  2. Change its network adapter from the LAN network to the DMZ virtual network.
  3. Power on Ubuntu-Target.

If using DHCP, renew the address:

sudo dhclient -r 2>/dev/null || true
sudo dhclient 2>/dev/null || true
ip -brief addr
ip route

If the dhclient command is not found, install it first:

sudo apt update -y
sudo apt install isc-dhcp-client

If no internet access is available after moving Ubuntu-Target to DMZ, in pfSense's WebGUI, go to "Firewall > Rules > DMZ". Add the following rule: 

Action: Pass
Interface: DMZ
Address Family: IPv4
Protocol: Any
Source: DMZ subnet
Destination: Any
Description: Allow DMZ to Internet

If using static addressing, use this example temporarily. Replace YOUR-INTERFACE with the actual interface name:

sudo ip addr flush dev YOUR-INTERFACE
sudo ip addr add 10.10.20.10/24 dev YOUR-INTERFACE
sudo ip link set YOUR-INTERFACE up
sudo ip route add default via 10.10.20.1

15. Troubleshooting Guide

ProblemLikely CauseFix
DMZ interface does not appear in pfSenseThird NIC not added or connected to wrong networkPower off pfSense, add NIC 3, connect it to the DMZ network, then boot.
Ubuntu-DMZ gets a LAN IP addressUbuntu still connected to LAN instead of DMZPower off Ubuntu and attach it to the DMZ virtual network.
Ubuntu-DMZ gets an unexpected IP addressVMware DHCP serving DMZ instead of pfSenseDisable VMware DHCP for DMZ or use static addressing.
Kali cannot reach DMZ web serverMissing LAN-to-DMZ rule, wrong IP, Apache stopped, or local firewall issueCheck pfSense LAN rules, Ubuntu IP, Apache status, and UFW.
DMZ can reach LAN when it should notDMZ block rule missing, below broad allow rule, or wrong destinationMove the DMZ-to-LAN block rule above broad allow rules and verify destination is LAN net.
Firewall logs do not show pass trafficPass rule logging may not be enabledEnable logging on the specific rule for lab evidence or document that pass traffic is not logged.
DMZ cannot access WAN for updatesNo DMZ outbound rule, DNS issue, or WAN problemCheck policy, rules, gateway, and DNS. Only allow outbound if instructor permits.
pfSense WebGUI unreachableKali not on LAN, wrong pfSense LAN IP, or rules changedUse pfSense console, verify LAN IP, and ensure Kali is on LAN network.
