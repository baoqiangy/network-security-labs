1. Lab Purpose

This lab introduces TCP and UDP ports from a practical network security perspective. Students will inspect which services are listening locally on Ubuntu-Target, generate controlled TCP and UDP traffic, scan the approved target from Kali-Lab, and explain why local service visibility and remote network visibility may differ.

Kali-Lab ───── Host-only / Private Network ───── Ubuntu-Target
Main goal: Students should understand that open ports represent reachable services and that network security work often begins by identifying what is listening, what is reachable, and what should or should not be exposed.

2. Learning Objectives

  1. Explain the difference between TCP and UDP at a basic operational level.
  2. Identify listening services on Linux using ss.
  3. Explain the meaning of local address, foreign address, port number, process, and protocol.
  4. Verify SSH as a TCP listening service on Ubuntu-Target.
  5. Start and test a temporary TCP web service on port 8080.
  6. Start and test a temporary UDP listener using nc when available.
  7. Use Nmap from Kali-Lab to compare local service state with remote visibility.
  8. Document service exposure using command output, screenshots, and tables.
  9. Explain OS-specific differences for Windows, Intel Mac, Apple Silicon Mac, Kali, and Ubuntu environments.

3. Lab Scenario

A junior security analyst has been asked to review an internal Ubuntu server. The first task is to determine what services are running and what ports are visible from another machine. The analyst must answer:

  1. Which TCP ports are listening on Ubuntu?
  2. Which UDP ports are listening or testable?
  3. Which services are reachable from Kali?
  4. Which tools show local service state?
  5. Which tools show remote network visibility?
  6. What should be documented in a service exposure report?

6. Safety Rules

  1. Run service and port tests only between Kali-Lab and Ubuntu-Target.
  2. Do not scan public systems, school networks, home routers, cloud systems, or classmates’ machines.
  3. Keep both VMs on the isolated host-only/private network.
  4. Do not use bridged networking.
  5. Do not run exploit tools, password attacks, denial-of-service tools, or unauthorized scans.
  6. Stop temporary services at the end of the lab.
  7. Do not submit screenshots containing passwords or sensitive personal information.

7. Key Concepts

TCP

TCP is connection-oriented. It establishes a session before data is exchanged. SSH and HTTP are common TCP-based services.

UDP

UDP is connectionless. It sends datagrams without establishing a session first. DNS commonly uses UDP, although DNS can also use TCP in some cases.

Listening Service

A listening service is a process waiting for network connections or messages on a specific port.

Part 4 — Inspect Local Listening Services on Ubuntu

On Ubuntu-Target:

sudo ss -tulpen

Save the output:

sudo ss -tulpen > ~/Lab2-1-PortsServices/service-info/ubuntu-listening-services-baseline.txt

Review the columns and identify:

  • Protocol: TCP or UDP
  • Local address
  • Port number
  • Process or program name
  • Whether the service listens on 127.0.0.1, 0.0.0.0, or the lab IP address

Required Screenshot 4: Baseline listening services on Ubuntu.

Part 5 — Verify SSH as a TCP Listening Service

On Ubuntu-Target:

sudo systemctl status ssh --no-pager
sudo ss -tulpen | grep ':22'

If SSH is not installed or running:

sudo apt update
sudo apt install -y openssh-server
sudo systemctl enable ssh
sudo systemctl start ssh

Save SSH evidence:

sudo systemctl status ssh --no-pager > ~/Lab2-1-PortsServices/service-info/ssh-status.txt
sudo ss -tulpen | grep ':22' > ~/Lab2-1-PortsServices/service-info/ssh-listening-port22.txt

Required Screenshot 5: SSH status and TCP port 22 listening on Ubuntu.

Part 6 — Test SSH from Kali

From Kali-Lab:

ssh YOUR-UBUNTU-USERNAME@UBUNTU-IP-ADDRESS

After login:

whoami
hostname
exit

Save a validation note on Kali:

echo "SSH TCP/22 was reachable from Kali to Ubuntu." > ~/Lab2-1-PortsServices/service-tests/ssh-port22-validation.txt

Required Screenshot 6: Successful SSH connection from Kali to Ubuntu.

Part 7 — Run a Safe Initial Nmap Scan from Kali

From Kali-Lab, scan only the approved Ubuntu-Target VM:

nmap -p 22,80,443,8080,9090,9091 UBUNTU-IP-ADDRESS

Save the output:

nmap -p 22,80,443,8080,9090,9091 UBUNTU-IP-ADDRESS \
-oN ~/Lab2-1-PortsServices/scan-results/nmap-initial-selected-ports.txt

Students should compare Nmap’s remote view with the local ss output on Ubuntu.

Required Screenshot 7: Initial Nmap selected-port scan from Kali.

Part 8 — Start a Temporary TCP Web Service on Ubuntu

On Ubuntu-Target, open a new terminal:

mkdir -p ~/Lab2-1-WebService
cd ~/Lab2-1-WebService
echo "Lab 2.1 TCP web service test" > index.html
python3 -m http.server 8080 --bind 0.0.0.0

Leave this terminal running. In a second Ubuntu terminal, verify the listener:

sudo ss -tulpen | grep ':8080'

Save evidence:

sudo ss -tulpen | grep ':8080' > ~/Lab2-1-PortsServices/listeners/tcp-8080-listener.txt

Required Screenshot 8: Temporary TCP web service listening on port 8080.

Part 9 — Test TCP Port 8080 from Kali

From Kali-Lab:

curl http://UBUNTU-IP-ADDRESS:8080

Expected output:

Lab 2.1 TCP web service test

Save the result:

curl http://UBUNTU-IP-ADDRESS:8080 > ~/Lab2-1-PortsServices/service-tests/http-8080-validation.txt

Run Nmap again:

nmap -p 22,8080 UBUNTU-IP-ADDRESS \
-oN ~/Lab2-1-PortsServices/scan-results/nmap-after-web-service.txt

Required Screenshot 9: curl success and Nmap showing port 8080 open.

Part 10 — Optional: Start a Temporary TCP Listener with Netcat

Check whether Netcat is available on Ubuntu:

nc -h 2>&1 | head

If nc is missing, install it only if the instructor allows temporary Internet access:

sudo apt update
sudo apt install -y netcat-openbsd

On Ubuntu-Target, start a TCP listener:

nc -l -p 9090

In a second Ubuntu terminal, verify:

sudo ss -tulpen | grep ':9090'

From Kali-Lab, connect and send a test message:

echo "Lab 2.1 TCP netcat test" | nc UBUNTU-IP-ADDRESS 9090

Stop the Ubuntu listener with Ctrl + C after receiving the message.

Optional Screenshot: Netcat TCP listener and client message.

Part 11 — Optional: Start a Temporary UDP Listener with Netcat

On Ubuntu-Target, start a UDP listener:

nc -u -l -p 9091

In a second Ubuntu terminal, verify UDP listener state:

sudo ss -ulpen | grep ':9091'

From Kali-Lab, send a UDP message:

echo "Lab 2.1 UDP netcat test" | nc -u UBUNTU-IP-ADDRESS 9091

Stop the Ubuntu UDP listener with Ctrl + C after receiving the message.

Optional Screenshot: Netcat UDP listener and client message.

UDP note: UDP testing may appear different from TCP because UDP does not establish a connection. Nmap UDP scans may also take longer or show less certain results.

Part 13 — Run a Final Remote Visibility Scan from Kali

From Kali-Lab, run a final selected-port scan:

nmap -sV -p 22,80,443,8080,9090,9091 UBUNTU-IP-ADDRESS \
-oN ~/Lab2-1-PortsServices/scan-results/nmap-final-service-visibility.txt

Expected result: no output for temporary services after they are stopped.

16. Instructor Notes

Core lesson: A port is not just a number; it represents a potential service exposure. Students should learn to verify services from both the local system view and the remote network view.

Recommended Teaching Sequence

  1. Review TCP and UDP conceptually.
  2. Show ss -tulpen on Ubuntu and explain each column.
  3. Verify SSH locally and remotely.
  4. Start a temporary web service on port 8080.
  5. Test it with curl and Nmap from Kali.
  6. Discuss local service state versus remote visibility.
  7. Optionally demonstrate Netcat TCP and UDP listeners.
  8. Stop temporary services and complete the report.

Recommended Short Version

  1. Run ss -tulpen on Ubuntu.
  2. Verify SSH on TCP/22.
  3. Scan selected ports from Kali with Nmap.
  4. Start Python web service on TCP/8080.
  5. Test with curl and Nmap.
  6. Stop temporary services.
  7. Complete service exposure summary and final report.
