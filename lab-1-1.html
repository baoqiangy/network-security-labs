Lab 1.1 — Virtual Network Security Lab Environment Setup

Network Security hands-on lab for setting up the isolated virtual environment used throughout the course. Students create Kali-Lab and Ubuntu-Target VMs, place them on a private virtual network, verify connectivity, enable SSH, and document OS-specific differences.

1. Lab Purpose

This lab establishes the virtual lab environment used throughout the Network Security course. Students will install or verify virtualization software, create two Linux virtual machines, connect them to an isolated virtual network, verify network connectivity, enable SSH access, and document their environment.

Kali-Lab ───── Host-only / Private Network ───── Ubuntu-Target
Main goal: Create a safe, isolated environment where Kali-Lab can communicate with Ubuntu-Target without exposing lab activity to public, school, work, or home networks.

2. Learning Objectives

  1. Explain why isolated virtual networks are used for cybersecurity labs.
  2. Create or verify a Kali Linux VM and an Ubuntu VM.
  3. Configure both VMs on the correct host-only or private virtual network.
  4. Identify IP addresses, routes, interfaces, and CPU architecture.
  5. Verify connectivity between Kali-Lab and Ubuntu-Target.
  6. Enable and test SSH access from Kali-Lab to Ubuntu-Target.
  7. Create a clean snapshot after the environment works.
  8. Document differences between Windows, Intel Mac, and Apple Silicon Mac environments.

3. Lab Scenario

You are preparing a controlled virtual network security lab. The environment must support future labs involving packet capture, scanning, firewall configuration, VPNs, vulnerability assessment, monitoring, and incident response. The setup must be isolated so students do not accidentally interact with real networks or other devices.

6. Safety Rules

  1. Use only the assigned Kali-Lab and Ubuntu-Target VMs.
  2. Do not use bridged networking for course labs unless explicitly instructed.
  3. Do not scan or test public systems, school networks, home routers, cloud systems, or classmates’ machines.
  4. Keep lab traffic on the isolated host-only/private virtual network.
  5. Do not perform password attacks, exploitation, denial-of-service, malware execution, or unauthorized scanning.
  6. Do not submit screenshots containing real passwords, private keys, or sensitive personal information.
  7. Create a clean snapshot once the environment is working.

7. Key Concepts

Host-Only / Private Network

A host-only or private virtual network allows VMs on the same host computer to communicate with each other while staying isolated from external networks.

Network ModeUse in This Course
Host-only / PrivateRecommended default for cybersecurity labs
NAT / Share with my MacUsed temporarily only when Internet access is needed for package installation
BridgedDo not use for these labs unless explicitly instructed

Kali-Lab

Kali-Lab is the security workstation. It will be used for command-line testing, Nmap scanning, SSH, tcpdump, Wireshark, evidence collection, and validation.

Ubuntu-Target

Ubuntu-Target is the target/server VM. It will be used for SSH access, temporary web services, firewall configuration, logs, VPN labs, hardening, and incident response practice.

Architecture

The VM architecture affects which installation image and packages can be used.

Architecture OutputMeaningCommon Host
x86_64Intel/AMD 64-bit Linux VMWindows PCs and Intel Macs
aarch64ARM64 Linux VMApple Silicon Macs

Required Screenshot 6: Show IP address, route, and architecture information for both VMs.

Part 7 — Test Network Connectivity

From Kali-Lab, ping Ubuntu-Target:

ping -c 4 UBUNTU-IP-ADDRESS

Example:

ping -c 4 192.168.56.102

From Ubuntu-Target, ping Kali-Lab:

ping -c 4 KALI-IP-ADDRESS

Expected result:

4 packets transmitted, 4 received, 0% packet loss

Required Screenshot 7: Show successful ping in both directions.

Part 8 — Install or Verify SSH on Ubuntu-Target

On Ubuntu-Target, check SSH status:

sudo systemctl status ssh --no-pager

If SSH is not installed:

sudo apt update
sudo apt install -y openssh-server

Enable and start SSH:

sudo systemctl enable ssh
sudo systemctl start ssh
sudo systemctl status ssh --no-pager
If Internet access is needed temporarily: Switch only the VM that needs packages to NAT or Share with my Mac, install the package, then switch it back to Host-only or Private to my Mac.

Required Screenshot 8: Show SSH installed and running on Ubuntu-Target.

Part 9 — Test SSH from Kali-Lab to Ubuntu-Target

From Kali-Lab:

hostname
ip addr

15. Troubleshooting Guide

ProblemLikely CauseFix
VM does not bootWrong ISO or architectureUse x86_64 images for Windows/Intel Mac; ARM64 images for Apple Silicon.
Kali cannot ping UbuntuVMs are on different virtual networksPut both VMs on VMnet1 Host-only or Private to my Mac.
No IP address appearsNetwork adapter disabled or DHCP issueCheck VM network adapter and restart networking or reboot the VM.
SSH connection refusedSSH server not installed or not running on UbuntuInstall/start openssh-server.
SSH times outWrong IP, network mismatch, or firewall issueVerify IP address, ping, and network mode.
Package install failsNo Internet access on host-only/private networkTemporarily switch to NAT or Share with my Mac, install packages, then switch back.
Apple Silicon VM image failsIntel/AMD image used on ARM hostUse ARM64-compatible Kali and Ubuntu images.
Evidence copy failsSSH issue, wrong username, or wrong pathVerify SSH login, username, IP address, and folder path.

16. Instructor Notes

Core lesson: A safe cybersecurity course begins with a controlled lab environment. Students must understand isolation, addressing, architecture, and evidence collection before performing later network security labs.

Recommended Teaching Sequence

  1. Explain why the lab must be isolated.
  2. Identify each student’s host OS and architecture.
  3. Create or verify Kali-Lab and Ubuntu-Target.
  4. Configure the correct virtual network.
  5. Boot both VMs and record IP information.
  6. Test ping in both directions.
  7. Enable SSH on Ubuntu.
  8. Test SSH from Kali.
  9. Create evidence folders and save network information.
  10. Create clean snapshots.

Recommended Short Version

  1. Configure both VMs on host-only/private network.
  2. Record IP addresses and architecture.
  3. Verify ping between VMs.
  4. Enable SSH on Ubuntu.
  5. SSH from Kali to Ubuntu.
  6. Create snapshots and submit evidence.
