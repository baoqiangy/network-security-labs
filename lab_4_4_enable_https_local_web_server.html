1. Lab Purpose

The purpose of this lab is to help students configure HTTPS on a local web server using a private CA-signed certificate. Students will install Apache on Ubuntu-Target, create a simple website, enable SSL/TLS, configure an HTTPS virtual host, test access from Kali-Lab, and verify that the certificate chains back to the private CA from Lab 4.3.

Main goal: Students should understand that HTTPS requires both encryption and trust: the server must present a valid certificate, and the client must trust the CA that issued it.

2. Learning Objectives

  1. Explain the purpose of HTTPS and TLS for protecting web traffic.
  2. Install or verify Apache on Ubuntu-Target.
  3. Create a simple local web page for HTTPS testing.
  4. Install a private CA-signed server certificate and private key on Ubuntu-Target.
  5. Enable Apache SSL support and configure an HTTPS virtual host.
  6. Test Apache configuration before restarting or reloading the service.
  7. Use curl and openssl s_client from Kali-Lab to verify HTTPS behavior.
  8. Compare untrusted, insecure, and trusted TLS verification results.
  9. Document certificate subject, issuer, SAN, validity, and verification status.
  10. Explain OS-specific differences for Windows, macOS, Kali, and Ubuntu.

3. Lab Scenario

A small organization has an internal web server running on Ubuntu. The organization wants the internal site to use HTTPS with a certificate issued by its private CA. Your job is to deploy the certificate, enable Apache HTTPS, and prove from Kali-Lab that the site is encrypted and trusted when the private CA certificate is supplied.

6. Safety Rules

  1. Configure HTTPS only on the assigned Ubuntu-Target VM.
  2. Use only lab-created certificates and keys.
  3. Do not submit private keys or passphrases.
  4. Do not expose the Apache server to public networks.
  5. Keep the virtual network host-only/private.
  6. Do not install the lab CA into production or personal trust stores unless explicitly instructed.
  7. Use -CAfile for trust testing instead of permanently trusting the lab CA system-wide.
  8. Back up Apache configuration before modifying it.
Important: This lab is for local HTTPS learning. The private key and private CA are lab assets and should never be used for real services.

7. Key Concepts

HTTPS

HTTPS is HTTP protected by TLS. It provides encrypted communication between a client and a web server and helps the client verify the server’s identity.

TLS Certificate

A TLS certificate binds a public key to a server identity, such as a DNS name or IP address. Clients verify the certificate chain and check whether the certificate matches the name used to connect.

Apache Virtual Host

An Apache virtual host defines how Apache serves a website. An HTTPS virtual host usually listens on port 443 and includes certificate and private-key paths.

Trust Anchor

A trust anchor is a CA certificate that the client trusts. In this lab, Kali trusts the private CA only when students provide the CA certificate using curl --cacert or openssl s_client -CAfile.

Part 8 — Install Certificate and Key on Ubuntu

On Ubuntu-Target:

sudo mkdir -p /etc/ssl/lab/certs
sudo mkdir -p /etc/ssl/lab/private

sudo cp ~/Lab4-4-HTTPS/cert-install/private_lab_root_ca.cert.pem /etc/ssl/lab/certs/

sudo chmod 600 /etc/ssl/lab/private/ubuntu_target_server.key.pem

Verify the certificate and key match:

sudo openssl x509 -noout -modulus -in /etc/ssl/lab/certs/ubuntu_target_server.cert.pem | openssl sha256
sudo openssl rsa -noout -modulus -in /etc/ssl/lab/private/ubuntu_target_server.key.pem | openssl sha256

The two hash values should match.

Part 11 — Create the HTTPS Virtual Host

On Ubuntu-Target:

sudo tee /etc/apache2/sites-available/labhttps-ssl.conf > /dev/null <<'EOF'
<VirtualHost *:443>
    ServerName ubuntu-target.local
    ServerAlias ubuntu-target

    DocumentRoot /var/www/labhttps

    SSLEngine on
    SSLCertificateFile /etc/ssl/lab/certs/ubuntu_target_server.cert.pem
    SSLCertificateKeyFile /etc/ssl/lab/private/ubuntu_target_server.key.pem

    Header always set X-Content-Type-Options "nosniff"
    Header always set X-Frame-Options "DENY"

    ErrorLog ${APACHE_LOG_DIR}/labhttps_ssl_error.log
    CustomLog ${APACHE_LOG_DIR}/labhttps_ssl_access.log combined

    <Directory /var/www/labhttps>
        Options -Indexes
        AllowOverride None
        Require all granted
    </Directory>
</VirtualHost>
EOF

sudo cat /etc/apache2/sites-available/labhttps-ssl.conf
