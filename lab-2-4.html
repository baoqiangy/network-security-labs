Lab 2.4 — SSH Service Testing and Log Review

Network Security hands-on lab based on the Lab 1.1 virtual environment. Students verify SSH service status, test secure remote access from Kali-Lab to Ubuntu-Target, generate safe authentication events, inspect SSH logs, and document remote-access security observations.

1. Lab Purpose

This lab introduces SSH as a secure remote-access service and teaches students how to verify service status, test connectivity, inspect listening ports, generate safe authentication events, and review logs for successful and failed access attempts.

Kali-Lab ───── Host-only / Private Network ───── Ubuntu-Target
Main goal: Students should understand SSH as both a secure administration tool and an important source of security monitoring evidence.

2. Learning Objectives

  1. Explain the purpose of SSH in secure remote administration.
  2. Verify that the SSH server is installed, enabled, and running on Ubuntu-Target.
  3. Use ss to confirm that SSH is listening on TCP port 22.
  4. Use SSH from Kali-Lab to connect to Ubuntu-Target.
  5. Generate safe successful and failed authentication events.
  6. Inspect SSH logs using journalctl and available authentication log files.
  7. Compare service status, port visibility, and authentication log evidence.
  8. Use Nmap to verify remote visibility of SSH from Kali-Lab.
  9. Document SSH testing evidence in tables and a short report.
  10. Explain OS-specific differences across Windows, Intel Mac, Apple Silicon Mac, Kali, and Ubuntu environments.

3. Lab Scenario

A security analyst is asked to verify whether SSH remote access is configured correctly on an internal Ubuntu server. The analyst must determine whether SSH is running, whether it is reachable from another system, whether login attempts are logged, and what security lessons can be learned from reviewing successful and failed access events.

6. Safety Rules

  1. Use only Kali-Lab and Ubuntu-Target.
  2. Do not attempt SSH logins against public systems, school systems, home routers, cloud systems, or classmates’ machines.
  3. Do not perform password attacks or brute-force testing.
  4. Generate only a small number of safe failed login events as instructed.
  5. Do not submit real passwords in screenshots, recordings, or reports.
  6. Keep both VMs on the isolated host-only/private network.
  7. Do not use bridged networking for this lab.
  8. Do not permanently weaken SSH configuration.

7. Key Concepts

SSH

Secure Shell, or SSH, provides encrypted remote access to a system. Administrators commonly use SSH to manage Linux servers securely.

Service Status vs. Network Reachability

QuestionToolWhat It Shows
Is the SSH service running locally?systemctl status sshService status on Ubuntu
Is TCP port 22 listening locally?ss -tulpenLocal socket and process information
Can another host see port 22?nmap -p 22Remote visibility from Kali
Can a user log in?ssh user@IPEnd-to-end authentication and remote access
Was the activity logged?journalctl or auth logsMonitoring and evidence trail

Authentication Events

SSH logs can show successful login events, failed login attempts, session openings, session closings, service starts, and service stops. These logs are important for security monitoring and incident response.

Common SSH Security Concerns

  • Weak passwords
  • Unnecessary exposure of TCP port 22
  • Root login enabled
  • Missing monitoring of failed login attempts
  • Overly broad network access

Part 4 — Install or Verify OpenSSH Server on Ubuntu

On Ubuntu-Target, check SSH service status:

sudo systemctl status ssh --no-pager

If SSH is not installed or the service is not found, install it. Use NAT or Share with my Mac temporarily only if Internet access is needed and allowed by the instructor:

sudo apt update
sudo apt install -y openssh-server

Enable and start SSH:

sudo systemctl enable ssh
sudo systemctl start ssh
sudo systemctl status ssh --no-pager

Save service status:

sudo systemctl status ssh --no-pager > ~/Lab2-4-SSH/service-info/ssh-service-status.txt

Required Screenshot 4: SSH service installed, enabled, and running on Ubuntu.

Part 5 — Verify SSH Is Listening on TCP Port 22

On Ubuntu-Target:

sudo ss -tulpen | grep ':22'

Save:

sudo ss -tulpen | grep ':22' > ~/Lab2-4-SSH/service-info/ssh-port22-listener.txt

Students should identify the protocol, local address, port, and process.

Required Screenshot 5: SSH listening on TCP port 22.

Part 6 — Review SSH Server Configuration Safely

On Ubuntu-Target, review key non-comment SSH settings:

sudo grep -E '^(Port|PermitRootLogin|PasswordAuthentication|PubkeyAuthentication|AllowUsers|DenyUsers)' /etc/ssh/sshd_config

If no output appears, that means these values may be using defaults or are commented out. Save both the filtered and full configuration:

sudo grep -E '^(Port|PermitRootLogin|PasswordAuthentication|PubkeyAuthentication|AllowUsers|DenyUsers)' /etc/ssh/sshd_config \
> ~/Lab2-4-SSH/config-review/sshd-config-key-settings.txt 2>&1 || true
sudo cp /etc/ssh/sshd_config ~/Lab2-4-SSH/config-review/sshd_config_copy.txt
Do not make permanent configuration changes in this lab unless instructed. This section is a review, not a hardening lab.

Required Screenshot 6: SSH configuration review evidence.

Part 7 — Test SSH from Kali to Ubuntu

From Kali-Lab:

ssh YOUR-UBUNTU-USERNAME@UBUNTU-IP-ADDRESS

When prompted, accept the host key only if the IP address is the correct Ubuntu-Target VM. After login:

whoami
hostname
ip -brief addr
exit

Save a validation note on Kali:

echo "SSH login from Kali to Ubuntu succeeded." > ~/Lab2-4-SSH/ssh-tests/successful-ssh-login-note.txt

Required Screenshot 7: Successful SSH login from Kali to Ubuntu.

Part 8 — Test SSH Verbose Mode

From Kali-Lab, run a verbose SSH connection test:

ssh -v YOUR-UBUNTU-USERNAME@UBUNTU-IP-ADDRESS

After login, type:

exit

Save verbose connection output:

ssh -v YOUR-UBUNTU-USERNAME@UBUNTU-IP-ADDRESS exit \
> ~/Lab2-4-SSH/ssh-tests/ssh-verbose-success.txt 2>&1

Students should identify evidence of connection setup, host key checking, authentication, and session close.

Required Screenshot 8: Verbose SSH connection output.

Part 9 — Run a Safe Nmap SSH Port Check

From Kali-Lab, scan only the approved Ubuntu-Target VM:

nmap -sV -p 22 UBUNTU-IP-ADDRESS

Save scan output:

nmap -sV -p 22 UBUNTU-IP-ADDRESS \
-oN ~/Lab2-4-SSH/scan-results/nmap-ssh-port22-service-check.txt

Required Screenshot 9: Nmap showing SSH service on TCP port 22.

Part 10 — Generate One Safe Failed SSH Login Event

Important: Do not perform brute-force testing. Generate only one or two safe failed attempts as instructed.

From Kali-Lab, intentionally use a wrong username or cancel a login attempt. Example using a clearly nonexistent username:

ssh fakeuser@UBUNTU-IP-ADDRESS

When prompted for a password, either enter one incorrect test password once or press Ctrl + C. Do not repeat many attempts.

Save a note:

echo "One safe failed SSH login test was generated using fakeuser." > ~/Lab2-4-SSH/ssh-tests/failed-login-test-note.txt

Required Screenshot 10: One safe failed SSH login attempt from Kali.

Part 11 — Review SSH Logs with journalctl

On Ubuntu-Target, review recent SSH service logs:

sudo journalctl -u ssh --no-pager | tail -100

Save the log review:

sudo journalctl -u ssh --no-pager | tail -150 > ~/Lab2-4-SSH/logs/ssh-journalctl-tail.txt

Create a filtered summary:

sudo journalctl -u ssh --no-pager | grep -Ei 'accepted|failed|invalid|session opened|session closed|started|stopped|listening' \
> ~/Lab2-4-SSH/logs/ssh-journalctl-filtered-summary.txt

Required Screenshot 11: SSH journal logs showing relevant SSH events.

Part 12 — Review Authentication Log Files If Available

On Ubuntu-Target, check for common authentication log files:

ls -lh /var/log/auth.log /var/log/secure 2>/dev/null || true

If /var/log/auth.log exists:

sudo tail -100 /var/log/auth.log
sudo grep -Ei 'sshd|accepted|failed|invalid|session opened|session closed' /var/log/auth.log | tail -100 \
> ~/Lab2-4-SSH/logs/auth-log-ssh-summary.txt

If /var/log/auth.log does not exist, document that journalctl was used instead:

echo "auth.log was not available; journalctl was used for SSH log review." \
> ~/Lab2-4-SSH/logs/auth-log-not-available-note.txt

15. Troubleshooting Guide

ProblemLikely CauseFix
Kali cannot ping UbuntuWrong IP or VMs are on different networksVerify ip -brief addr and correct VMware network mode.
systemctl status ssh says service not foundOpenSSH server is not installedInstall openssh-server if instructor permits.
SSH service is inactiveService is stoppedRun sudo systemctl start ssh.
Port 22 is not listeningSSH not running or configured for another portCheck service status and /etc/ssh/sshd_config.
SSH connection refusedSSH service not listeningStart SSH and verify with ss.
SSH connection times outWrong IP, firewall issue, or network mismatchVerify ping, route, and network mode first.
Login failsWrong username or passwordVerify Ubuntu username and use the correct password without showing it in screenshots.
Logs do not show expected SSH entriesWrong log source or not enough recent activityGenerate one test login and check journalctl -u ssh.
/var/log/auth.log does not existLogging configuration differs by distributionUse journalctl -u ssh and document that auth.log was unavailable.
Evidence copy failsWrong username, IP, path, or SSH issueVerify SSH login and folder path before using scp.
