1. Lab Purpose

The purpose of this lab is to introduce Linux logging and auditing as a foundation for security monitoring. Students will use Ubuntu-Target as the monitored system and Kali-Lab as a remote test client. They will identify important log locations, generate safe authentication and file-access events, configure auditd rules, search audit records, and create a short security log review report.

Main goal: Students should understand that logs are evidence. Effective security monitoring requires knowing where logs are stored, what events are captured, how to search them, and how to explain what they mean.

2. Learning Objectives

  1. Explain the role of Linux logs in security monitoring and incident investigation.
  2. Identify common Linux log sources such as journalctl, /var/log/auth.log, and service logs.
  3. Install, start, and verify auditd on Ubuntu-Target.
  4. Create temporary audit rules using auditctl.
  5. Create persistent audit rules in /etc/audit/rules.d/.
  6. Generate safe authentication, sudo, file-change, and service events.
  7. Search audit records using ausearch and summarize events using aureport.
  8. Compare system logs, authentication logs, and audit logs.
  9. Document findings, timestamps, event sources, and security meaning.
  10. Explain logging differences across Ubuntu, Kali, Red Hat-based Linux, Windows, and macOS.

3. Lab Scenario

You are a security analyst responsible for reviewing activity on an internal Ubuntu server. Your supervisor wants evidence of authentication activity, privileged command use, changes to sensitive files, service events, and auditd rule behavior. You must configure safe local audit rules, generate controlled events, and prepare a log review report that explains what each event means.

6. Safety Rules

  1. Generate only controlled events in the local lab environment.
  2. Do not attempt brute force login testing. One or two intentionally failed login tests are enough if assigned.
  3. Do not collect or expose real passwords, tokens, private keys, or sensitive personal data.
  4. Do not modify critical system files except for lab-approved audit configuration files.
  5. Use a lab directory for file-change testing.
  6. Back up audit rules before changing persistent audit configuration.
  7. Do not apply immutable audit rules unless instructed; immutable rules can require reboot to change.
  8. Do not run this lab on production systems.
Important: Logs may contain usernames, hostnames, IP addresses, and command history. Treat log data as sensitive evidence and submit only lab-related artifacts.

7. Key Concepts

System Logs

System logs record operating system and service events. On systemd-based Linux systems, journalctl provides a central way to query the system journal.

Authentication Logs

Authentication logs record login attempts, SSH activity, sudo use, and privilege-related events. On Ubuntu/Debian systems, many authentication events appear in /var/log/auth.log.

Auditd

auditd is the Linux audit daemon. It records audit events based on kernel audit rules, including file access, command execution, identity changes, and security-relevant system activity.

Audit Rules

Audit rules define what auditd should watch. Rules can monitor specific files or directories and can assign a key such as lab71_watch to make searches easier.

Evidence Quality

Good log evidence includes the event time, source host/IP, user, command or file involved, event result, and explanation of why the event matters.

Part 8 — Add Temporary Audit Rules

On Ubuntu-Target, add temporary audit rules for this lab. These rules remain until reboot or audit rules are reloaded.

sudo auditctl -w /opt/lab71-audit -p wa -k lab71_dir_changes
sudo auditctl -w /etc/passwd -p wa -k lab71_identity_files
sudo auditctl -w /etc/group -p wa -k lab71_identity_files
sudo auditctl -w /etc/sudoers -p r -k lab71_sudoers_read
sudo auditctl -l | tee ~/Lab7-1-LoggingAuditd/auditd/audit-rules-temporary.txt

Rule meaning:

  • -w defines the watched file or directory.
  • -p wa watches write and attribute changes.
  • -p r watches read access.
  • -k assigns a search key.

sudo auditctl -s
sudo ausearch -k lab71_dir_changes

15. Troubleshooting Guide

ProblemLikely CauseFix
auditctl not foundauditd package is not installedInstall auditd if instructor permits.
auditd service will not startPackage issue, kernel audit issue, or service errorCheck systemctl status auditd and journalctl -xe.
No audit events foundRule not loaded, wrong key, or no event generatedRun sudo auditctl -l, regenerate event, and search by correct key.
/var/log/auth.log not foundDifferent distribution or logging configurationUse journalctl or distribution-specific auth log such as /var/log/secure.
SSH events not visibleWrong service name, SSH not installed, or no event generatedCheck journalctl -u ssh, journalctl -u sshd, and SSH service status.
Permission denied reading logsLogs require elevated privilegeUse sudo for protected logs.
Persistent rules do not loadSyntax error in rule fileCheck /etc/audit/rules.d/lab71.rules and run sudo augenrules --load.
Rules cannot be changedAudit rules may be immutableDo not set immutable mode in this lab. If already immutable, reboot may be required.
UFW logs missingUFW disabled or logging disabledDocument as not applicable or enable only if instructor permits.
