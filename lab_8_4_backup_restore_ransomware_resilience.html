1. Lab Purpose

The purpose of this lab is to teach backup and restore as a practical cybersecurity control. Students will create a small set of dummy files, produce verified backups, simulate data loss in a controlled ransomware-style exercise, restore files, validate integrity with checksums, and document resilience gaps.

Main goal: Students should understand that backup only becomes a security control when it is tested, protected, documented, and recoverable within acceptable time and data-loss limits.

2. Learning Objectives

  1. Explain how backup and restore support ransomware resilience.
  2. Define recovery point objective, recovery time objective, and backup integrity.
  3. Create a lab-only dataset of dummy business files.
  4. Create full and incremental-style backups using local Linux tools.
  5. Generate and verify file checksums before and after backup.
  6. Simulate ransomware impact safely without malware, encryption routines, persistence, or real data exposure.
  7. Restore files from a backup archive and compare restored files to the original baseline.
  8. Measure restore time and estimate data loss.
  9. Document backup risks such as untested backups, writable backups, missing offsite copies, and weak retention.
  10. Create a prioritized ransomware resilience improvement plan.
  11. Compare backup and restore practices across Linux, Windows, macOS, cloud, and virtual machine environments.

3. Lab Scenario

A small organization stores important operational files on an internal endpoint. Management is worried that ransomware could corrupt or destroy local files. You are asked to test whether the organization can recover from a simulated ransomware-style data-loss event. The simulation must use only dummy files inside a clearly marked lab directory. You must prove that backup data is recoverable and that restored files match the original data.

6. Safety Rules

  1. Use only dummy files created for this lab.
  2. Run the simulation only inside ~/Lab8-4-BackupResilience/live-data.
  3. Do not run malware, ransomware samples, exploit tools, or real encryption routines.
  4. Do not target real documents, user home directories, shared drives, cloud folders, or production systems.
  5. Do not add persistence, spreading behavior, privilege escalation, evasion, or destructive system commands.
  6. Do not delete backup archives during the simulation.
  7. Do not mount real external drives unless the instructor explicitly assigns it.
  8. Back up files before running any simulation step.
  9. Clearly label all files as lab simulation data.
  10. Stop immediately if a command points outside the lab directory.
Important: This lab does not teach ransomware development. It uses a harmless file-impact simulation on dummy files only so students can practice backup verification and restoration.

7. Key Concepts

Backup

A backup is a copy of data that can be used to restore information after accidental deletion, corruption, hardware failure, or cyberattack.

Restore

A restore is the process of recovering data from backup media or backup archives and validating that recovered data is usable.

Ransomware Resilience

Ransomware resilience is the ability to continue or recover operations after data is encrypted, corrupted, deleted, or held hostage. Resilience includes prevention, detection, backup protection, restore testing, and incident response planning.

Recovery Point Objective

Recovery Point Objective, or RPO, is the maximum acceptable amount of data loss measured in time. For example, an RPO of 24 hours means the organization can tolerate losing up to one day of data.

Recovery Time Objective

Recovery Time Objective, or RTO, is the maximum acceptable amount of time required to restore service or data after an incident.

Backup Integrity

Backup integrity means the backup is complete, uncorrupted, and restorable. Checksums and restore tests help prove integrity.

3-2-1 Backup Rule

The 3-2-1 rule recommends at least three copies of data, on two different media or storage types, with one copy offsite or isolated.

Part 9 — Create a Safe Ransomware-Style Simulation Script

Safety reminder: This script is not ransomware. It does not encrypt files, spread, persist, hide, evade detection, or target anything outside the lab directory. It only renames and replaces contents of dummy files inside live-data.

Create the simulation script on Ubuntu-Target:

cat > ~/Lab8-4-BackupResilience/simulation/safe-ransomware-impact-simulation.sh <<'EOF'
#!/usr/bin/env bash
set -euo pipefail

LAB_ROOT="$HOME/Lab8-4-BackupResilience"
TARGET_DIR="$LAB_ROOT/live-data"
LOG_FILE="$LAB_ROOT/logs/safe-ransomware-impact-simulation.log"

echo "=== Lab 8.4 Safe Ransomware Impact Simulation ===" | tee "$LOG_FILE"
echo "Start time: $(date)" | tee -a "$LOG_FILE"
echo "Target directory: $TARGET_DIR" | tee -a "$LOG_FILE"

case "$TARGET_DIR" in
  "$HOME"/Lab8-4-BackupResilience/live-data) ;;
  *)
    echo "ERROR: Target directory is outside the approved lab path. Exiting." | tee -a "$LOG_FILE"
    exit 1
    ;;
esac

if [[ ! -d "$TARGET_DIR" ]]; then
  echo "ERROR: Target directory does not exist. Exiting." | tee -a "$LOG_FILE"
  exit 1
fi

find "$TARGET_DIR" -type f | sort | while read -r file; do
  echo "Simulating impact on: $file" | tee -a "$LOG_FILE"
  cp "$file" "$file.pre-simulation-copy"
  {
    echo "SIMULATED RANSOMWARE IMPACT - LAB ONLY"
    echo "Original file content replaced for backup/restore practice."
    echo "Timestamp: $(date)"
    echo "This is not malware and contains no encryption routine."
  } > "$file"
  mv "$file" "$file.simulated_locked"
done

find "$TARGET_DIR" -type f | sort | tee "$LAB_ROOT/logs/live-data-after-simulation-file-list.txt"
echo "End time: $(date)" | tee -a "$LOG_FILE"
echo "Simulation complete. Restore from backup to recover clean files." | tee -a "$LOG_FILE"
EOF

chmod +x ~/Lab8-4-BackupResilience/simulation/safe-ransomware-impact-simulation.sh
sed -n '1,120p' ~/Lab8-4-BackupResilience/simulation/safe-ransomware-impact-simulation.sh

11. OS-Specific Differences

Ubuntu / Linux

Linux systems commonly use tools such as tar, rsync, sha256sum, find, diff, and cron/systemd timers for backup workflows.

tar -czf backup.tar.gz data/
rsync -av data/ backup-copy/
sha256sum file.txt
find data -type f
diff -qr data restore-data

Kali Linux

Kali can be used as an evidence collection system or remote test workstation, but backup operations should be performed on the endpoint being protected.

scp -r USER@UBUNTU-IP:/home/USER/Lab8-4-BackupResilience ./ubuntu-evidence
ssh USER@UBUNTU-IP "find ~/Lab8-4-BackupResilience -type f | wc -l"

Virtual Machine Snapshots

VM snapshots can help in labs, but snapshots are not a complete backup strategy. They may depend on the same host storage and may not provide long-term, offsite, or immutable recovery.
