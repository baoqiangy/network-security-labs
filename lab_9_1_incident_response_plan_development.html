1. Lab Purpose

The purpose of this lab is to help students create a practical incident response plan that could be used by a small organization. Students will organize the plan around preparation, detection and analysis, containment, eradication, recovery, communication, evidence handling, and lessons learned.

Main goal: Students should understand that incident response is not only a technical activity. It also requires roles, decision authority, communication, documentation, evidence handling, recovery planning, and post-incident improvement.

2. Learning Objectives

  1. Explain the purpose of an incident response plan.
  2. Identify common incident response phases and what happens during each phase.
  3. Define incident response roles and responsibilities for a small organization.
  4. Create an incident severity classification model.
  5. Develop escalation and notification procedures.
  6. Create evidence handling and chain-of-custody templates.
  7. Create communication templates for technical and non-technical stakeholders.
  8. Develop response playbooks for common incidents such as phishing, malware, ransomware, lost device, unauthorized access, and service outage.
  9. Connect response planning to logging, SIEM alerts, endpoint hardening, and backup recovery.
  10. Run a short tabletop exercise and document lessons learned.
  11. Create a final incident response plan with appendices and supporting forms.
  12. Compare incident response planning needs across Linux, Windows, macOS, cloud, network, and hybrid environments.

3. Lab Scenario

You are part of a small organization that has recently improved its network security lab environment. The organization now has Linux endpoints, basic logging, optional Wazuh monitoring, optional Suricata alerts, endpoint hardening checklists, and backup/restore procedures. However, it does not yet have a written incident response plan. You have been assigned to develop a plan that tells staff what to do when a security incident occurs.

6. Safety Rules

  1. Do not perform real attacks to test the incident response plan.
  2. Use only simulated incidents, instructor-provided alert cards, or evidence from previous safe labs.
  3. Do not include real passwords, tokens, private keys, confidential data, or personal information in the plan.
  4. Do not contact real external parties such as law enforcement, regulators, customers, or vendors during the lab.
  5. Do not claim that a simulated event is a real incident.
  6. Clearly label all scenarios, forms, and reports as classroom/lab exercises.
  7. Do not make production configuration changes as part of this planning lab.
  8. Keep evidence handling examples limited to dummy lab files, sample logs, or redacted screenshots.
Important: This lab is a planning and tabletop exercise. Students should practice decision-making and documentation without causing real system impact.

7. Key Concepts

Incident

An incident is an event that compromises, threatens, or violates the confidentiality, integrity, or availability of systems, data, users, or business operations.

Incident Response Plan

An incident response plan is a documented procedure that defines how an organization detects, reports, analyzes, contains, eradicates, recovers from, and learns from security incidents.

Severity

Severity describes the impact and urgency of an incident. A severity model helps teams decide when to escalate and what response resources are needed.

Containment

Containment limits the spread or impact of an incident. Examples include isolating a host, disabling an account, blocking an IP address, or restricting network access.

Eradication

Eradication removes the cause of the incident, such as malware, unauthorized accounts, vulnerable services, or malicious configuration changes.

Recovery

Recovery restores affected systems and data to normal operations while validating that the threat has been removed and controls are functioning.

Chain of Custody

Chain of custody is documentation that tracks evidence collection, handling, transfer, storage, and access so that evidence remains trustworthy.

11. Environment Differences

Linux / Ubuntu Endpoints

Linux incident response often relies on authentication logs, systemd journal, auditd, file permissions, process listings, service status, and package/update records.

journalctl --since "24 hours ago" --no-pager
sudo tail -n 100 /var/log/auth.log
sudo ausearch -k KEY -i
sudo ss -tulpen
systemctl --type=service --state=running

Kali Linux

Kali may be used as an analyst workstation for controlled testing, evidence collection, and reporting. It should not be used to attack systems as part of this planning lab.

scp -r USER@HOST:/path/to/evidence ./incident-evidence
sha256sum evidence-file
nmap -sT -Pn -p 22,80,443 LAB-HOST-IP

16. Instructor Notes

Core lesson: This lab should help students move from technical security controls to operational readiness. A good incident response plan tells people what to do, who decides, how evidence is handled, how recovery is validated, and how the organization improves afterward.
  1. Encourage students to use a fictional organization if privacy is a concern.
  2. Require role-based contacts rather than real phone numbers or personal data.
  3. Use prior labs as evidence sources: Wazuh alerts, Suricata alerts, backup restore results, endpoint hardening findings, and audit logs.
  4. Grade the plan for usability, not only length.
  5. Have students test the plan using a short tabletop exercise.
  6. Emphasize that incident response requires technical, management, communication, legal, and recovery coordination.
